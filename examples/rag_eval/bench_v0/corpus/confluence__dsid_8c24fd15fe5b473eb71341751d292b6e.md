# Adr Hosted Rollout Cohorting And Traffic Splitting

Source type: confluence
Document ID: dsid_8c24fd15fe5b473eb71341751d292b6e
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
ADR: Hosted rollout cohorting and percentage traffic splitting (deterministic + region behavior)

## Summary
This ADR defines how Redwood Hosted assigns requests to **baseline** vs **canary** cohorts during a hosted canary rollout, and how the configured canary percentage is enforced across regions. The decision prioritizes:
- Deterministic, reproducible cohort assignment (especially across retries)
- Sticky behavior when customers provide stable identifiers
- Minimal coordination/state across regions and data planes
- Observability clarity when effective serving differs from intended cohort

## Status
**Published** (effective for Hosted canary rollouts GA).

## Context / problem
Hosted canary rollouts require percentage-based traffic splitting on a Hosted route (e.g., 1% → 5% → 25% → 50% → 100%) while maintaining:
- Consistent cohorting so a given end user/conversation does not “flip” unpredictably
- Correctness under retries, streaming, and regional failover
- Clear metrics for baseline vs canary cohorts (including when the system must fall back)

Constraints:
- Data plane enforcement must be fast and operate without a central “assignment service.”
- Region routing and model fallback already exist; cohorting must compose cleanly.
- Customers may or may not provide stable end-user identifiers.

## Decision
### 1) Deterministic cohorting function
Cohort assignment is computed as a pure function:

**cohort = f(route_id, rollout_id, cohort_key)**

Where:
- `route_id` is the canonical Hosted route identifier
- `rollout_id` is the rollout instance identifier (changes per rollout)
- `cohort_key` is a stable string chosen per request (see selection rules below)

Implementation:
- Hash function: **xxHash64** (or equivalent stable 64-bit non-cryptographic hash) over UTF-8 bytes
- Input format (must be stable and unambiguous):
  - `"v1|" + route_id + "|" + rollout_id + "|" + cohort_key`
- Convert hash → bucket:
  - `bucket = hash64 % 10000` (0–9999)
- Compare to configured canary percentage:
  - `is_canary = bucket < (canary_percent * 100)`
  - Example: 5% → threshold 500

Rationale:
- Pure function = deterministic across restarts, regions, and retries (given same `cohort_key`).
- 10,000 buckets provides 0.01% granularity while remaining simple.
- Including `rollout_id` avoids “stickiness” across separate rollouts (a new rollout should re-sample cohorts).

### 2) Cohort key selection rules (stickiness)
The data plane selects `cohort_key` using the following precedence order:

1. **Explicit Cohort Key header (preferred):**
   - `X-Redwood-Cohort-Key` (customer-provided stable identifier)
   - Recommended values: end_user_id, account_id, conversation_id, or a stable hash thereof

2. **Idempotency key (recommended for retries/streaming reconnects):**
   - `Idempotency-Key` (if present)
   - Intended to remain identical across client retries and stream reconnect attempts

3. **Request ID (best-effort fallback):**
   - `X-Request-Id` (if present)
   - If generated by Redwood, it is unique per attempt and does **not** provide end-user stickiness; it only makes repeated internal handling of the same request deterministic.

4. **Non-sticky fallback (last resort):**
   - A per-request generated UUID (ensures correct aggregate percentage but no per-user stickiness)

Console/Docs requirements:
- Console wizard and docs MUST recommend using `X-Redwood-Cohort-Key` or `Idempotency-Key` for best results.
- Console must warn when a customer’s observed request mix suggests low stickiness (optional; informational).

### 3) Where cohorting is enforced
Cohorting and split enforcement occur in the **Smart Routing / data plane** after route resolution and policy checks, and before final model selection.

Order (simplified):
1. Authenticate request / resolve tenant + route
2. Evaluate policy eligibility for rollout (RBAC, guardrails, rollout state)
3. Select region per existing routing policies (latency/capacity/affinity)
4. **Compute cohort (baseline/canary) in the chosen region’s data plane**
5. Select model version/config for the cohort
6. Apply existing fallback policies if the chosen target is unavailable

### 4) How configured percentages apply across regions
Configured canary percentage is enforced **independently per region**, using the same deterministic function.

Meaning:
- The system does **not** attempt to maintain an exact global canary percentage across all regions combined.
- Each region will converge to the configured percentage for traffic that is routed to that region.

Rationale:
- Avoids cross-region coordination/state and reduces blast radius.
- Aligns canary metrics to the region where traffic was actually served.
- Prevents “global percentage drift” caused by sudden regional routing shifts.

Known consequence:
- If a single end user is routed to different regions on different requests (e.g., due to region failover or customer not using region affinity), they may land in different cohorts because the cohorting is evaluated after region selection and because the available model set can differ by region.

Mitigation:
- Customers who require strict consistency should use region pinning/affinity (where supported) and provide a stable `X-Redwood-Cohort-Key`.

### 5) Handling retries, streaming, and internal re-attempts
#### Client retries
- If the client supplies the same `Idempotency-Key` across retries, cohort assignment remains stable.
- If no idempotency/cohort key is provided, cohort assignment may change between retries (expected).

#### Internal retries (within a request attempt)
- Internal re-attempts must preserve the computed cohort for the request.
- Data plane should compute cohort once and propagate it via internal request context to downstream services.

#### Streaming
- Streaming responses are treated as a single request attempt. Cohort is computed at stream start.
- If a client reconnects and expects consistent behavior, it must reuse `Idempotency-Key` (and ideally `X-Redwood-Cohort-Key`).

### 6) Fallback behavior and “effective cohort” telemetry
If the intended cohort target cannot be served (e.g., canary model unavailable in a region, or hard failure triggers fallback), the system may serve from an alternate target.

Definitions:
- **intended_cohort:** baseline | canary (result of cohort function)
- **effective_target:** the model/version/config actually used after fallback
- **effective_cohort:** baseline | canary, derived from `effective_target` mapping

Telemetry requirements (must be emitted on every request):
- `rollout_id` (when applicable)
- `intended_cohort` (baseline/canary)
- `effective_cohort` (baseline/canary)
- `fallback_reason` (empty or one of: canary_unavailable, canary_error, capacity, policy_denied, region_failover, other)

Customer-facing metrics views:
- Default charts compare **effective_cohort** baseline vs canary (what was actually served).
- Secondary diagnostics show intended vs effective mismatch counts (to detect “phantom canary”).

Safety rule:
- If the rollout is in a state requiring canary availability guarantees (e.g., higher steps), policy may deny progressing if `canary_unavailable` is non-zero in any eligible region for a configured lookback window (guardrail; tracked separately).

## Options considered
### Option A: Pure random assignment per request
- Pros: simple; aggregate percentage correct
- Cons: not sticky; terrible for user experience comparisons; retries change cohort; noisy metrics
- Rejected.

### Option B: Central assignment service (store user→cohort mappings)
- Pros: strong stickiness; globally consistent
- Cons: adds new stateful service; latency and availability risks; heavy operational overhead; complicated privacy considerations
- Rejected for GA.

### Option C: Deterministic hashing with stable key (this decision)
- Pros: no state; deterministic; low latency; supports stickiness when customers provide keys
- Cons: stickiness depends on key hygiene; cross-region can vary depending on routing behavior
- Accepted.

### Option D: Enforce percentage globally across regions
- Pros: global percentage invariance
- Cons: requires coordination/state or cross-region sampling; complex under routing shifts
- Rejected for GA. (Can be revisited if we introduce a global routing coordinator.)

## Detailed specification
### Cohort key normalization
To avoid accidental cohort churn:
- Use the cohort key exactly as received (no trimming/case folding), except:
  - Reject keys longer than **256 bytes** (return validation error: `cohort_key_too_long`)
  - Reject keys containing NUL bytes
- Do not log raw cohort keys (treat as potentially sensitive). If needed for debugging, log a truncated hash.

### Percent precision and step changes
- Percent is stored as a decimal with two decimal places (0.00–100.00).
- Changing percent changes the threshold; users move cohorts deterministically based on their bucket.
- This behavior is desired: when going from 5%→25%, the additional 20% is a deterministic superset.

### Exclusions / bypass
Certain traffic must not participate in rollouts:
- Health checks, internal synthetic probes, and system-to-system calls used for SLO verification.

Mechanism:
- Internal callers set `X-Redwood-Bypass-Rollout: true` (restricted to Redwood internal auth).
- If bypassed, cohorting is skipped and the baseline target is selected; telemetry marks `rollout_bypassed=true`.

## Consequences
### Positive
- Deterministic and reproducible cohorting without introducing stateful infrastructure.
- Supports real-world safe rollouts (sticky cohorts) when customers follow recommended headers.
- Clear observability story with intended vs effective cohort tracking.

### Negative / tradeoffs
- Stickiness is only as good as the provided key; without it, customer experience may vary.
- Regional independence means some users may move cohorts if they move regions.
- Must be careful with metric cardinality (rollout_id + cohort tags); see telemetry guidelines.

## Implementation notes (non-normative)
- Data plane should compute cohort as early as possible after route resolution, but after any bypass/policy deny checks.
- Keep the hash implementation identical across languages (gateway, edge proxies, and routing services). Add a golden test vector suite.
- Add integration tests to validate:
  - Stable assignment across retries with same idempotency key
  - Step increases are monotonic superset of canary population
  - Fallback produces intended/effective mismatch telemetry

## Security and privacy considerations
- Treat `X-Redwood-Cohort-Key` as customer data; do not store raw values.
- Audit logs should record that a rollout split is enabled and its configured percentage, but must not include per-request cohort keys.

## Rollout / migration plan
- Beta: allow only for tenants with feature flag enabled; strongly recommend cohort key usage.
- GA: document cohort key + idempotency best practices; ship dashboard panels that surface intended vs effective mismatch.

## Open questions / follow-ups
1. Should we expose a first-class `cohort_key` field in the API body (in addition to a header) for SDK ergonomics?
2. Should we allow customers to choose “global cohorting before region selection” mode for strict stickiness (likely requires a stable region selection key)?
3. What are the default guardrails for blocking step progression if effective canary rate is far below configured percent due to availability?
