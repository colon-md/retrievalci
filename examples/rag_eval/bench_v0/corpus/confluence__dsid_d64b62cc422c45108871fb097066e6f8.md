ADR-007: Standard KMS Client Wrapper and Failure Policy

## Status
Published.

## Summary
We will standardize all Key Management Service (KMS/HSM-backed) calls in the Redwood monorepo behind a shared **KMS client wrapper** with:
- Explicit, consistent **timeouts** and **retry budgets** (with jitter/backoff).
- A canonical **error taxonomy** used across logs, metrics, traces, and user-facing/operator-facing messages.
- A shared **circuit breaker** and **backpressure** policy to prevent retry storms and cascading failures when KMS is degraded.
- Strict **caching constraints** and safe defaults appropriate for envelope encryption.

This ADR defines the wrapper API shape, failure-handling rules, observability requirements, and non-goals.

## Context / Problem
Redwood’s Hosted, Dedicated, and Private deployments all integrate with customer-managed KMS/HSM for envelope encryption flows (e.g., DEK generation/encryption, DEK decryption, key metadata lookups).

Prior to this ADR, KMS interactions differed across services (control plane vs data plane), leading to:
- Inconsistent timeouts and SDK default retries (often too aggressive) causing **tail latency** and **retry amplification**.
- Non-uniform error handling (e.g., throttling treated as generic 500s) which increased time-to-triage.
- Missing or inconsistent telemetry (no shared tag set; inconsistent metric names).
- Confusing operator experiences (unclear “AccessDenied” vs wrong key ARN vs key disabled).

We also experienced a near-miss retry storm during a regional timeout spike where client retries amplified load on KMS and increased 5xx rates.

## Goals
1) Provide a single, reviewed integration point for KMS calls with safe defaults.
2) Ensure predictable latency behavior via explicit timeouts + bounded retries.
3) Improve triage and post-incident analysis via a canonical taxonomy + telemetry.
4) Prevent cascading failures via circuit breakers and backpressure.
5) Establish guardrails so services cannot silently override safety-critical defaults without an explicit, audited override.

## Non-goals
- Introducing new cryptographic primitives or changing algorithms.
- Building a “KMS proxy” service or sidecar as part of this initiative.
- Implementing a universal key rotation workflow (see ADR-008).
- Persistently caching plaintext key material or storing decrypted DEKs outside process memory.

## Decision
### Decision 1: All KMS calls must go through a shared wrapper library
We will implement and mandate a shared wrapper library (e.g., `redwood/kmsclient`) used by:
- Control plane services that manage key references and encryption configuration.
- Data plane services that decrypt/encrypt envelope payloads.
- Private installer/validator components that perform preflight KMS checks.

Direct usage of cloud provider SDK KMS clients in production services is disallowed unless an exception is approved (see Key Management Standard / exceptions process).

### Decision 2: Establish a canonical error taxonomy
All errors surfaced by the wrapper are mapped into a small set of canonical error categories with stable semantics.

#### Canonical error categories
| Category | Examples | Retry? | Severity / Notes |
|---|---|---|---|
| `KMS_ACCESS_DENIED` | IAM denies, missing `Decrypt`, wrong principal | No | Actionable operator error; include missing permission and key identifier if available. |
| `KMS_NOT_FOUND` | key ID/ARN/alias not found, wrong region, deleted key | No | Actionable; often config error. |
| `KMS_INVALID_CIPHERTEXT` | ciphertext invalid, wrong key version, corrupted payload | No | Usually indicates key mismatch or data corruption; triggers incident workflow depending on scope. |
| `KMS_KEY_DISABLED` | key disabled/pending deletion | No | Actionable; can block all decrypt; escalate to customer/security immediately. |
| `KMS_THROTTLED` | provider quota exceeded, `ThrottlingException`, 429 | Yes (bounded) | Retry with exponential backoff + jitter; enforce throttle budget. |
| `KMS_TIMEOUT` | request deadline exceeded | Yes (bounded) | Retry only within total budget; feed into circuit breaker. |
| `KMS_TRANSIENT` | 5xx, connection reset, temporary DNS | Yes (bounded) | Retry with jitter; feed into circuit breaker. |
| `KMS_MISCONFIGURED` | invalid key ARN format, unsupported region, missing endpoint in Private | No | Fail fast at startup/preflight where possible. |
| `KMS_UNKNOWN` | unclassified | Maybe (bounded) | Treated as transient only if clearly network/5xx; otherwise no retry. |

#### Requirements for error payloads
The wrapper must attach structured fields consistently:
- `kms.provider` (aws|gcp|azure)
- `kms.operation` (Decrypt|Encrypt|GenerateDataKey|GetKey|GetKeyPolicy|…)
- `kms.key_ref` (redacted/hashed if customer-sensitive; never log plaintext secrets)
- `kms.region`
- `error.category` (canonical)
- `error.code` (provider code if present)
- `retry.attempt` and `retry.exhausted` (boolean)
- `circuit.state` (closed|open|half_open)

### Decision 3: Timeouts and retry budgets are explicit and bounded
We will not rely on cloud SDK defaults.

#### Default timeout and retry policy (safe defaults)
All values below are defaults; services may tighten them (stricter) without approval. Loosening them requires an explicit override with Security + SRE review.

| Operation | Default per-attempt timeout | Max attempts | Total time budget (cap) | Retry on |
|---|---:|---:|---:|---|
| `Decrypt` | 750ms | 2 | 1.5s | `KMS_THROTTLED`, `KMS_TIMEOUT`, `KMS_TRANSIENT` |
| `Encrypt` | 750ms | 2 | 1.5s | `KMS_THROTTLED`, `KMS_TIMEOUT`, `KMS_TRANSIENT` |
| `GenerateDataKey` | 1.0s | 2 | 2.0s | `KMS_THROTTLED`, `KMS_TIMEOUT`, `KMS_TRANSIENT` |
| `GetKeyMetadata` / key lookup | 500ms | 2 | 1.0s | `KMS_THROTTLED`, `KMS_TIMEOUT`, `KMS_TRANSIENT` |

Backoff: exponential with full jitter.
- Initial delay: 50ms
- Max delay: 300ms
- Jitter: required
- Respect `Retry-After` where provided (bounded by total time budget)

Retry is explicitly disabled for non-retryable categories (`KMS_ACCESS_DENIED`, `KMS_NOT_FOUND`, `KMS_INVALID_CIPHERTEXT`, `KMS_KEY_DISABLED`, `KMS_MISCONFIGURED`).

### Decision 4: Add a shared circuit breaker and backpressure policy
We will implement a circuit breaker per (provider, region, operation) with a conservative default to prevent request storms.

#### Circuit breaker semantics
- **Closed:** normal operation; collect rolling metrics.
- **Open:** fail fast with a typed error (`KMS_DEPENDENCY_UNAVAILABLE`) without calling KMS.
- **Half-open:** allow a limited number of probe requests to test recovery.

#### Default breaker thresholds
(Subject to tuning during rollout; changes require SRE review.)
- Rolling window: 60s
- Minimum request volume before evaluation: 50
- Open breaker if either:
  - Failure rate >= 20% for retryable categories (`KMS_TIMEOUT`, `KMS_TRANSIENT`, `KMS_THROTTLED`) OR
  - Consecutive timeouts >= 25
- Open duration: 30s (then half-open)
- Half-open probe rate: max 1 req/s per key (provider/region/operation)

#### Backpressure (throttle budget)
When `KMS_THROTTLED` is observed, the wrapper will apply a token-bucket-style throttle budget per (provider, region, operation) to avoid saturating KMS under load spikes. If budget is exhausted, the wrapper fails fast with `KMS_THROTTLED` and `retry.exhausted=true` rather than attempting additional retries.

### Decision 5: Caching is allowed only for metadata and short-lived in-memory key material, with strict constraints
Because we are using envelope encryption, caching can reduce KMS load but must not expand the blast radius of key compromise.

#### Allowed caches
1) **Key metadata cache** (safe):
- Examples: alias → key ARN, key state, supported algorithms
- TTL default: 1 hour
- Negative cache TTL (not found): 60 seconds
- Must be invalidated on rotation/config change events where applicable.

2) **In-memory DEK/plaintext key material cache** (restricted):
- Allowed only for plaintext DEKs that are already present in memory for request processing.
- TTL default: 5 minutes (configurable downwards)
- Max entries: service-specific but must be bounded (default 10,000 entries) with LRU eviction.
- Scope: in-process only; never persisted; never shared cross-host.
- Must not include raw customer identifiers in cache keys (use hashed stable identifiers).
- Must be disable-able via config in regulated environments.

#### Disallowed caches
- Persisting plaintext DEKs (disk, database, distributed cache).
- Long-lived plaintext caches (> 15 minutes) without a security exception.
- Caching ciphertext blobs in a way that bypasses key version validation or rotation safeguards.

## Rationale
- **Consistency + safety:** A single wrapper ensures safe defaults and reduces per-team re-implementation risk.
- **Reliability:** Explicit deadlines and a circuit breaker prevent KMS incidents from becoming platform-wide outages due to retry amplification.
- **Observability:** Canonical taxonomy and stable fields enable dashboards/alerts/runbooks that work across all deployments.
- **Enterprise readiness:** Actionable errors and documented behavior are required for regulated customers and security reviews.

## Detailed Design Notes
### Wrapper API (language-agnostic contract)
The wrapper must expose an interface roughly equivalent to:
- `Encrypt(ctx, keyRef, plaintext, opts)`
- `Decrypt(ctx, keyRef, ciphertext, opts)`
- `GenerateDataKey(ctx, keyRef, opts)`
- `GetKeyMetadata(ctx, keyRef, opts)`

Where `opts` includes:
- `deadline` (optional; defaults applied if absent)
- `idempotency_key` (where supported/needed)
- `cache_policy` (metadata-only; allow/deny in-memory DEK cache)
- `telemetry_context` (service name, environment, deployment mode)

### Telemetry requirements
Metrics (names are illustrative; final names must follow observability standards):
- `kms.client.request_latency_ms` (histogram) with tags: provider, region, operation, deployment_mode, service
- `kms.client.errors_total` (counter) with tags: provider, region, operation, error.category, error.code
- `kms.client.retries_total` (counter) with tags: provider, region, operation
- `kms.client.circuit_open_total` (counter) with tags: provider, region, operation
- `kms.client.cache_hits_total` / `kms.client.cache_misses_total` (counter) with tags: cache_type

Tracing:
- A span per KMS call with attributes matching the structured fields listed above.
- Errors must be recorded on the span with canonical error category.

Logging:
- Log at WARN for retryable failures when retries are exhausted.
- Log at ERROR for non-retryable categories and for circuit breaker opens.
- Never log plaintext key material or decrypted DEKs.

### Failure policy by component type
We explicitly distinguish between:
- **Data-plane decrypt path:** If decrypt fails, the request fails. The wrapper’s job is to fail quickly and consistently under KMS degradation, not to silently bypass encryption.
- **Control-plane metadata/config:** Prefer fail fast on misconfiguration (preflight validation, startup checks). For transient KMS errors, allow bounded retries but do not block entire control-plane reconciliation loops beyond the global time budget.
- **Installer/preflight (Private):** Must provide actionable diagnostics and an exportable validation report. Retries may be higher in preflight mode (since it is operator-driven), but still bounded.

## Alternatives Considered
1) **Do nothing / per-service KMS clients**
- Rejected: inconsistent timeouts/retries/telemetry; repeated mistakes; higher incident risk.

2) **Rely on cloud provider SDK default retries/timeouts**
- Rejected: defaults vary by language/provider; often tuned for throughput not tail latency; risk of retry storms.

3) **Service mesh / network-level retries**
- Rejected: cannot classify KMS-specific error codes well; may retry non-idempotent operations incorrectly; obscures metrics.

4) **Central KMS proxy service**
- Deferred: could improve policy enforcement and caching, but adds a new dependency and availability concerns; out of scope for this quarter.

5) **Adopt a third-party library as-is (e.g., generic KMS wrappers)**
- Rejected: insufficient support for Redwood’s deployment modes, telemetry conventions, and failure policy requirements; would still require significant customization.

## Consequences
### Positive
- Reduced risk of cascading failures during KMS degradation.
- Better on-call ergonomics and faster triage due to consistent taxonomy and fields.
- Easier security reviews and documentation consistency across Hosted/Dedicated/Private.

### Negative / Tradeoffs
- Slight increase in shared-library coupling; wrapper changes require careful rollout.
- Conservative time budgets may increase request failures during severe KMS degradation; this is an explicit tradeoff to protect overall platform availability and prevent retry amplification.

## Rollout Plan
1) Implement wrapper with feature flags and dual instrumentation.
2) Migrate highest-risk call sites first (data-plane decrypt/encrypt).
3) Enforce guardrails:
   - Lints / codeowners for direct SDK usage.
   - CI checks that require wrapper usage in approved services.
4) Roll out by environment (dev → staging → prod) with SRE-defined canary criteria:
   - No increase in 5xx beyond error budget threshold.
   - No increase in P99 latency beyond agreed guardrails.
   - Circuit breaker open rate within expected bounds.
5) Post-rollout review and tuning of thresholds.

## Security & Compliance Notes
- Logging must avoid customer-sensitive identifiers where feasible; prefer hashed or redacted key references.
- Overrides that loosen retry/timeouts or extend key material caching require documented approval (Security + SRE) and must be auditable.
- This ADR aligns with the Key Management Standard; any deviations require an exception request.

## Open Questions
- Provider-specific nuances for Azure Key Vault throttling and recommended retry headers (tracked in implementation tickets).
- Whether certain metadata calls can be consolidated and cached more aggressively without compromising correctness during rotation (coordinate with ADR-008).
