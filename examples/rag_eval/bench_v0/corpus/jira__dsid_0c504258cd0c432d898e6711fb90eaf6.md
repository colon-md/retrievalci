Northwind Commerce (Dedicated): transient p95 latency spike during case study measurement/interview window

Issue summary
- Northwind Commerce noticed an elevated p95 latency window on their Dedicated cluster while Marketing/SE were collecting performance numbers for a planned public case study.
- The spike was transient (~18 minutes) but would materially change any "single-window" p95 figure if captured during that interval.

Customer impact
- Endpoint: /v1/chat.completions (streaming + non-streaming)
- Impact: p95 increased from baseline ~430–520ms to ~1.6–2.1s; p99 peaked ~3.8s.
- Error rate: no sustained 5xx increase; brief uptick in 429s due to queue protection (max in-flight).
- User-visible symptoms: slower first-token latency on some requests; a few retries observed by Northwind.

Context (why ticket was opened)
- This was surfaced during case study preparation (interview + metric capture). We need to:
  1) confirm whether the spike was a platform issue vs workload change,
  2) ensure mitigations prevent recurrence during future measurement windows,
  3) document the event so published metrics are measured over an appropriate window (e.g., 7–30 days) and/or exclude known maintenance events.

What we checked
- Workload: No evidence of a step-function traffic increase from Northwind; request rate remained within typical burst bounds (+8% vs same weekday baseline).
- Model/version: No model rollout; pinned model version unchanged.
- Platform: K8s events showed a single GPU node drain/replace in the Dedicated pool coincident with the spike.

Investigation notes (most likely cause)
- Root cause: transient capacity reduction during a node drain (kernel security patch) combined with conservative surge settings for the deployment, leading to fewer ready replicas for ~10–15 minutes.
- Contributing factors:
  - Warm-up for the replacement node took longer than expected (image pull + cache warm), increasing request queue depth.
  - Autoscaler reacted, but scale-up lag (GPU provisioning) meant tail latency increased before capacity stabilized.

Mitigation applied
- Short-term (same day):
  - Increased min_replicas for Northwind Dedicated deployment from 6 -> 8 during business hours.
  - Adjusted pod disruption budget to reduce simultaneous unready capacity during drains.
  - Coordinated with Infra to pause non-urgent maintenance on Northwind’s Dedicated pool during the case study measurement week.
- Follow-up hardening:
  - Added an alert on "ready replicas < desired" + queue depth correlation for Dedicated clusters.
  - Documented recommended measurement window guidance for Marketing/SE: use rolling 7-day p95 (excluding maintenance/drain events) or present a range with a footnote.

Customer comms
- We informed Northwind that we identified a transient capacity event (node drain) and applied safeguards to reduce the chance of recurrence. No action required on their side.

Resolution
- Latency returned to baseline at 2025-10-09 17:10 UTC.
- Preventative changes are in place (min replicas + PDB + maintenance pause).
- Guidance provided to Marketing/SE to avoid quoting p95 captured during the impacted interval; use an agreed measurement window for publication.

Next steps / action items
1) Infra: validate node warm-up time on Dedicated pools; ensure image pre-pull + model cache warm path for replacement nodes (linked PR: redwood-inference/infra#19344).
2) Runtime: add "pre-warm" hook to reduce first-request penalty after replica replacement (linked PR: redwood-runtime#8821).
3) Support Ops: add a checklist item for case study metric capture: confirm no maintenance/drain events in the selected interval.
