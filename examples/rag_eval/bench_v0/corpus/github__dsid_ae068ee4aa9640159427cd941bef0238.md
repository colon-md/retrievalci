# Pr 18421 Multipart File Validation Limits

Source type: github
Document ID: dsid_ae068ee4aa9640159427cd941bef0238
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
add multipart/form-data handling, strict content-type validation, and payload limit enforcement for API tool/file inputs

Motivation: users integrating tool/function calling and file uploads via the OpenAI-compatibility endpoints were sending a wide variety of content-types and very large multipart requests which caused inconsistent runtime behavior, high memory pressure, and unexpected acceptance of unsupported payloads. This PR introduces a focused, backwards-compatible implementation for multipart/form-data parsing on the OpenAI-compatibility path, strict Content-Type validation for tool/file inputs, and deterministic payload limits both per-file and per-request. Summary of changes: 1) Add a streaming-safe multipart parser that yields parts rather than buffering entire requests; 2) Validate Content-Type of each part (whitelist: text/*, image/*, application/json, application/octet-stream with explicit size checks); 3) Enforce configurable limits: max_file_size (default 10MiB), max_total_request_size (default 50MiB), max_parts_count (default 20). These defaults are conservative and documented; they can be overridden at tenant or route level via API config. 4) Hook validation into the OpenAI-compat layer so existing client integrations get predictable errors (400/413) with structured error bodies when inputs are rejected. 5) Add detailed unit and integration tests and end-to-end tests for streaming uploads and for the dedicated Dedicated/Private ingress paths. Backwards compatibility notes: this does not change the non-multipart JSON body semantics. Existing users who relied on accepting arbitrary large file uploads should update their tenant config or use Redwood Dedicated with higher quotas. Rollout plan: feature-flagged behind feature.enable_multipart_input (off by default); staged rollout to internal dogfood tenants, then gradual roll to 10%, 50%, 100% via config rollout. Checklist: - [x] Streaming-safe multipart parser implemented - [x] Content-Type whitelist + validation errors - [x] Per-file and per-request size enforcement + configurable defaults - [x] OpenAI-compat error mapping and structured error shapes - [x] Tests: unit, integration, e2e - [x] Docs: compatibility notes and config guide - [x] Telemetry: counters for rejected parts and throttled requests - [x] Migration & rollout plan included Testing notes: the multipart parser uses a small constant memory footprint for each active part and yields part metadata to the handler; large parts trigger a streaming drain with early termination (413 response) to avoid OOM. Integration tests exercise combination of binary and text parts, wrong content-types, and boundary cases around the configured limits. Telemetry: added metrics api.multipart.rejected_content_type, api.multipart.rejected_size, api.multipart.accepted_parts and a hist for part sizes. Implementation details: - parser: runtime/multipart/stream_parser.py - validator: api/validation/multipart_validator.py - compat layer: api/openai_compat/input_adapter.py - config: config/defaults/multipart.yaml - docs: docs/openai-compat/multipart.md - tests: tests/unit/test_multipart_parser.py, tests/integration/test_openai_compat_multipart.py Error shape examples (structured output): when a part is rejected we return 413 with a JSON body {"error":{"type":"request_too_large","message":"part exceeds max_file_size","meta":{"part_name":"file","limit_bytes":10485760}}} and 400 for unsupported content-types with type unsupported_media_type. Observability: trace spans added around parse->validate->route dispatch to help pinpoint latency regressions.
