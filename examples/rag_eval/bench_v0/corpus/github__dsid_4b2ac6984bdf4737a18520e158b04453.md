# Pr 19422 Fix Throughput Metric Double Counting Streaming

Source type: github
Document ID: dsid_4b2ac6984bdf4737a18520e158b04453
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Fix throughput metric double-counting for streaming responses

Problem: Dedicated throughput SLO reporting was overcounting tokens for streaming requests. We were incrementing `deployment_tokens_generated_total` per-stream chunk (delta tokens) and then incrementing again on stream completion using the final `usage.completion_tokens` value. For non-streaming, the completion-path increment is correct; for streaming, it results in ~2x token totals (shape/model dependent) and skews derived tokens/sec in the SLO evaluator + dashboards.

Customer/ops impact: This primarily affects Dedicated throughput SLO compliance tracking (tokens/sec floor) and internal saturation indicators. It also contributed to customer confusion when comparing Console throughput graphs vs request-level `usage` totals (see SUP-18510). Billing exports are not impacted (billing uses metering events, not the SLO throughput counters).

Fix: Make streaming accounting single-source-of-truth by only emitting token increments from the streaming path, and skipping the completion-path increment when `streaming=true`. Additionally, guard against duplicate “final chunk” emission by tracking `last_emitted_completion_tokens` on the stream session and only emitting positive deltas.

What changed: - `api-gateway`: adjust token accounting so the completion handler does not increment throughput counters for streaming requests; the counter is incremented only via the stream chunk emitter. - `telemetry`: add explicit `streaming` + `accounting_source` labels to aid debugging (chunk vs completion). - tests: add unit tests for streaming lifecycle (N chunks + final usage) asserting exact token totals; add regression test covering the previously failing scenario where both chunk deltas and final usage were emitted.

Rollout notes: This changes the magnitude of streaming throughput counters immediately on deploy (expected ~-50% for affected traffic). SRE/Obs have been pinged to annotate the Dedicated throughput SLO dashboard and ignore pre/post comparisons for 24h.

Follow-ups (not in this PR): 1) Backfill/annotation for SLO reports that span the deploy window (SRE to decide). 2) Consider moving all throughput SLO computations to request-level usage events once metering latency is acceptable.

Validation: - Replayed a production trace set in staging (mixed streaming/non-streaming) and confirmed `deployment_tokens_generated_total` matches sum(usage.completion_tokens) within expected variance from retries/cancels. - Verified SLO evaluator output for a known Dedicated deployment no longer shows >2x spikes during high streaming traffic.
Sean Gallagher (SRE): Can you confirm this won’t undercount if the client disconnects before the final usage payload? Author: Yes—stream chunks already carry deltas. If the stream ends early, we count what was actually emitted; the completion-path increment was previously attempting to “catch up” but was double-counting when the final usage did arrive.
Allison Grant (Observability): Please add a label so we can distinguish stream chunk vs completion in case we have to debug again. Author: Added `accounting_source={chunk|completion}` and ensured completion source is only used for non-streaming.
Sergio Costa (Obs): Dashboard query expects `deployment_tokens_generated_total` without extra labels; will this break it? Author: Counter name unchanged; new labels are additive with stable defaults. Queries aggregating by deployment_id remain valid.
Owen Phillips (Support Eng): SUP-18510 mentions a mismatch between Console and customer-side totals; are we aligning with `usage.completion_tokens`? Author: Yes. For streaming, the sum of chunk deltas equals the final usage total (modulo cancels). We now avoid adding the final total on top.
