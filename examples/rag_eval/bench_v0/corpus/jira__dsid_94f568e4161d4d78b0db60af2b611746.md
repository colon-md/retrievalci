# Sup 48377 Canary Traffic Split Not Matching Config

Source type: jira
Document ID: dsid_94f568e4161d4d78b0db60af2b611746
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Hosted canary rollout: traffic split in us-east not matching Console config (canary receiving ~25% when set to 10%)

Issue summary
Customer reports that the canary cohort is receiving substantially more traffic than configured for a Hosted route rollout. Console shows canary set to 10% (after being reduced from 25%), but observed canary traffic remains ~22–27% in us-east.

Impact
- Customer sees increased cost/latency regression exposure compared to intended 10% canary.
- Customer is a beta design partner; they are using canary rollouts to validate a new quantization profile.

Customer report (via Support)
- Route: /v1/chat/completions (Hosted)
- Workspace/tenant: northstar-prod
- Rollout name/id (from customer screenshot): rll_9f3c2d3a
- Baseline: llama-3.1-70b-instruct v2025-04-03
- Canary: llama-3.1-70b-instruct v2025-04-16 (q8_0)
- Traffic schedule:
  - Step 1: 1% (ok)
  - Step 2: 5% (ok)
  - Step 3: 25% (ok)
  - Step 4: reduced to 10% (intended rollback in step) — mismatch started here

Environment
- prod, region: us-east
- Customer notes: eu-west appears closer to expected (not fully confirmed), but the majority of their traffic is us-east.

Evidence / reproduction
Customer provided:
1) Request logs showing response header X-Redwood-Rollout-Cohort=canary for ~1/4 of requests while Console shows 10%.
2) A sample of request IDs from a 10-minute window (2025-04-18 16:05–16:15 UTC) where cohort assignment was canary unexpectedly.

Support verification
Pulled routing decision logs in us-east for tenant northstar-prod and rollout rll_9f3c2d3a:
- router_decision fields observed:
  - rollout_id=rll_9f3c2d3a
  - configured_canary_percent=25 (unexpected)
  - console_step_target_percent=10 (expected)
  - cohort=canary
- Log example (redacted):
  - ts=2025-04-18T16:11:03Z request_id=req_01HV... tenant=northstar-prod route=/v1/chat/completions region=us-east rollout_id=rll_9f3c2d3a configured_canary_percent=25 hash_bucket=17 cohort=canary

Initial hypothesis
Smart Routing in us-east is using a stale rollout config (still at 25%) after a step change to 10% was applied in Console/control plane.

Immediate mitigation
- If confirmed stale config on a subset of router pods, restart/roll router deployment in us-east OR force a full config resync.
- Customer can also pause rollout to 0% canary as a safety measure (but report indicates Console changes are not reflecting, so may not be sufficient without router refresh).

Requested from customer
- Exact time they changed from 25% -> 10%
- Whether they use a consistent cohort key (e.g., user_id) vs relying on request_id (for sanity; not expected to cause systematic 2.5x increase, but helpful)

Internal notes
This resembles the beta incident in #incidents where a watch stream drop caused rollout config updates not to propagate to a subset of router pods in us-east. See related incident and upcoming router watchdog/resync change.

Next actions
- (Eng) Verify config watch health + last-applied rollout version on us-east router pods.
- (SRE) Check for watch lag / dropped stream in us-east around 15:40–16:00 UTC.
- (Support) Provide customer with mitigation steps and expected timeline.
