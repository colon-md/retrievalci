# Pr 22044 Savings Estimator V0 For Batching And Caching

Source type: github
Document ID: dsid_fee94d81c032494cbfce8094468f0973
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Savings estimator v0 for batching + caching suggestions (cost + latency deltas)

Implements estimator v0 for the Optimize Suggestions beta (batching/caching only). Given a route/model aggregate window, the estimator outputs (a) estimated cost delta (USD/day + percent), (b) estimated latency deltas (p50/p95), and (c) confidence inputs that feed the risk scoring model (handled in a follow-up PR).

Motivation: The Console Suggestions view needs conservative, explainable estimates that are stable over time and can be recomputed on a schedule without per-request overhead.

Scope: (1) Savings model for two suggestion types: enable/raise continuous batching and enable prefix/KV caching, (2) assumption config + overrides, (3) unit tests + golden fixtures, (4) serialization to the suggestions record used by the Suggestions API.

Non-goals: quantization recommendations, per-tenant guaranteed savings, and automatic application. This PR does not change rollout behavior; it only computes estimates.

Implementation notes (high level): Estimator is implemented as a pure function over telemetry aggregates (qps, token in/out stats, baseline p50/p95, cache metrics when available) plus a bounded set of assumptions (token price, batch efficiency curve, cache hit uplift caps, and safety clamps). We bias toward under-estimating savings and over-estimating latency impact.

Key behaviors: - If telemetry is missing or sample size is below thresholds, estimator returns `insufficient_data` with reason codes (used by UI empty states). - Estimated savings are capped by `max_savings_pct` (default 35% for batching, 45% for caching) to avoid inflated outputs early in beta. - Latency deltas are computed as a function of concurrency + batch size proposal with a monotonic clamp so we never claim improved p95 from batching when baseline is already low-latency.

Config: Added `optimize.estimator.v0.*` config keys with sane defaults and per-env overrides. Assumptions are logged into the suggestion payload (for UI 'Assumptions' disclosure) but do not include any secret values.

Example output (illustrative): `estimated_savings_usd_per_day: 128.40`, `estimated_savings_pct: 0.18`, `latency_delta_ms_p50: +12`, `latency_delta_ms_p95: +48`, `confidence: medium`, `reasons: ["window_qps_ok", "seq_len_stable", "cache_hit_missing_using_prior"]`.

Testing: Unit tests cover: boundary conditions (low QPS, missing cache metrics), regression fixtures for representative workloads, and stability tests to ensure small telemetry changes don’t cause large estimate jumps. Added a small property-based test to validate clamps/monotonicity.

Rollout: No runtime impact on request path. Used by the suggestions compute job only. Observability and UI wiring are in separate PRs.
