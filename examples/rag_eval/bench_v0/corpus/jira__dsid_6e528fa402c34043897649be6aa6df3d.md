# Sup 2892 Fallback Model Not Triggering During Overload

Source type: jira
Document ID: dsid_6e528fa402c34043897649be6aa6df3d
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Fallback model not triggering during overload (customer receives 503 OVERLOADED instead of routed fallback)

Issue summary
Banyan Search reports that their configured fallback model is not being used during overload events. Instead, requests fail with 503 and Redwood overload error codes.

Customer expectation
When primary model (rw-llama-3.1-70b-instruct) is overloaded, Redwood should route to configured fallback (rw-llama-3.1-8b-instruct) per their tenant policy.

Impact
- Customer sees elevated failed requests during traffic spikes rather than degraded (lower quality) responses.
- Affects production search assistant experience and increases client retry pressure.

Environment
- Hosted API
- Region: us-east
- Route: /v1/chat/completions
- Streaming: mostly enabled

Customer configuration (as described by customer)
- Primary: rw-llama-3.1-70b-instruct
- Fallback: rw-llama-3.1-8b-instruct
- Trigger condition: overload (customer referenced 'circuit breaker open')

What the customer is seeing (excerpt)
- HTTP 503
- Error payload (representative):
  code: OVERLOADED
  message: "Service temporarily overloaded. Please retry."
  correlation_id: rwcorr_01JEMNQZJ5J9P2G6E3R9K8V6M2
  retry_after: (present for some responses; missing for some per SUP-2860)

Steps to reproduce (internal)
1) Identify Banyan tenant_id and active routing policy.
2) Induce overload condition (or replay from perf-canary) where gateway admission control sheds requests.
3) Confirm whether routing attempts fallback model when overload is detected.

Initial hypothesis
Fallback routing is only attempted when overload is detected at routing/runtime stages. When overload occurs at the gateway admission layer (breaker open / shed), the request is rejected before routing executes, so no fallback can occur. Additional suspicion: loop-prevention logic may be suppressing fallback when the request already contains an attempted-model chain header.

Requested outcome
- Confirm whether fallback is expected to work under gateway-level shedding.
- If not currently supported, provide clear customer guidance and (if safe) enable fallback-on-gateway-overload for this tenant as an exception/feature flag.
- Ensure Retry-After is consistently present on overload errors.

That makes the router think it already tried the fallback and stops.
2025-12-10 09:30 PT — Ravi Menon (Fallback routing architect): Root cause seems twofold:
(1) Most failures happen pre-routing (gateway admission sheds) => fallback not possible.
(2) In the subset that reaches routing, a header bookkeeping issue causes loop-prevention to incorrectly block fallback.
Proposed fix: clear attempted_models when the failure is classified as 'admission_shed' vs 'model_attempted'. We can hotfix routing to only append attempted_models after a backend attempt is made.
