# Int 460811 Edge Switch Arp Proxy Ip Duplication

Source type: jira
Document ID: dsid_59cb294da8964683bb165a3bb615df71
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Edge switch ARP proxying leads to VPN client IP duplication and intermittent routing to wrong host

Issue summary: Several users reporting intermittent loss of access to internal services while connected to corporate VPN. Symptoms include duplicate IP conflict alerts on endpoints, frequent VPN re-auth events, and traffic appearing to be routed to a different on-prem host (service 10.23.5.12). Impact: ~8% of VPN sessions in us-west over the last 48h experienced short blackholes (30-90s) impacting git pushes, internal CI jobs, and ssh sessions. Scope: affects users on Wi-Fi SSID 'Redwood-Corp' behind specific edge switches in building SF2 rack 7.
1) Connect laptop to 'Redwood-Corp' Wi-Fi in SF2 building (tested on macOS 13 and Ubuntu 22.04)
2) Establish VPN session to redwood-vpn-usw.redwood.net (WireGuard-based corporate VPN)
3) Trigger continuous ping to an internal host (ping -i 0.2 10.23.5.12)
4) Observe intermittent duplicate IP warnings on client and packets forwarded to an unexpected MAC on the L2 domain
5) Reproduce frequency increases when there is a short burst of multicast/mDNS traffic on the same AP's radio
Client syslog excerpt: 'kernel: arp: IP address 10.23.8.45 claimed by 00:11:22:33:44:55 (conflict)'
Switch (edge-sw-7) log: 'proxy-arp conflict detected for 10.23.8.45, source 00:66:77:88:99:AA bridging-port 5'
Packet capture (pcap) snippet: ARP reply for 10.23.8.45 with two different source MACs within 120ms
VPN server metrics: client rekey count spike (client 10.23.8.45) correlated to timestamp 2026-03-12T14:12Z
Internal service (10.23.5.12) reported TCP reset spikes aligning with client blackholes
Hypothesis 1: Edge switch proxy-arp configured for legacy NAT path is responding to ARP for VPN-assigned addresses from an overlapping DHCP pool, causing two active ARP entries on the LAN.
Hypothesis 2: A misconfigured VXLAN/bridge on edge-sw-7 is advertising ARP for the VPN subnet into the campus L2 segment when multicast bursts occur.
What we tried: - Disabled proxy-arp on edge-sw-7 port-channel to AP group temporarily (no immediate reduction). - Isolated a suspect AP by moving a test client to a different AP (blackhole persisted only when client served by AP-7A). - Captured pcap on AP, switch uplink, and VPN server concurrently for correlated timestamps (pcaps saved to /mnt/logs/INT-460811/pcaps/).
Preliminary find: ARP 'gratuitous' replies with VPN client IP are being injected by the switch's 'proxy-arp' feature only when the AP's multicast queue exceeds threshold. The switch returns ARP for the same IP with a different MAC that matches a spare management interface in rack 7 (00:66:77:88:99:AA).
Intermittent connectivity loss for internal developer workflows (git/ssh/CI). Approximately 8% of VPN sessions in the SF2 us-west site affected in the window; potential for data-transfer retries and CI flakiness. No customer-facing outage but notable developer productivity hit.
1) Placed affected edge switch (edge-sw-7) in monitoring schedule and enabled debug logging for proxy-arp events.
2) Temporarily increased DHCP lease affinity for VPN pool for clients in SF2 to reduce the window for IP reuse.
3) Asked campus networking team to disable 'proxy-arp' global feature on a staging switch and observe (ongoing).
4) Notified platform reliability and CI owners of potential re-run needs for impacted jobs (Slack #incidents and #platform-alerts).

Switch firmware (1.2.3) bug caused proxy-arp behavior under high multicast load leading to ARP advertisements for VPN-assigned addresses. This created ephemeral duplicate IP assignments on the L2 domain and caused VPN clients to experience routing blackholes and rekeys.
Mitigation applied: 'proxy-arp disable' set on edge-sw-7 AP VLANs during maintenance window 2026-03-14 07:00-07:15.
Temporary workaround: increased DHCP lease affinity and moved high-multicast APs to separate RF profile to reduce multicast bursts.
Permanent fix: planned firmware upgrade to 1.2.6 across SF2 edge switches (racks 4-9) and config baseline change to disable proxy-arp for VPN/DHCP subnets. Vendor TAC provided patch notes and recommended upgrade path.
