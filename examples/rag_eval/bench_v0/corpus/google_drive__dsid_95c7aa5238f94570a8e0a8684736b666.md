Dedicated Burst Capacity — Post-beta retro notes (SRE)

Doc goal
Capture what we learned from the Dedicated Burst Capacity beta (May 2025): what broke, which guardrails were too strict/too lax, what we changed during beta, and what must be completed before GA.

What went well (keep)
1) Global safety cap + circuit breaker concept worked in practice
- We hit the cap once in staging and once “near miss” in prod beta; breaker prevented runaway neighbor impact.
- The playbook of “reduce caps -> drain queues -> disable burst for lowest tiers first” is operationally sound.

2) Denial reason codes improved triage materially
- Support/CS could distinguish “cap reached” vs “pool saturated” vs “cooldown” vs “telemetry stale”.
- Routing logs + reason codes made it clear where to look first.

3) Dashboards were mostly the right primitives
- Saturation, queue depth, admit/deny rates, and p95 deltas by tenant were enough to debug 80% of issues.
- SLO gate signal (non-burst tenant latency regression) is a good north star even if thresholds need tuning.

4) Rollout gating/feature flags prevented blast radius
- Per-region allow list + per-tenant enablement meant we could pause EU-West issues without impacting US-East.


What broke / painful incidents
A) Telemetry lag caused unfair denials + occasional starvation
- Symptom: a few tenants saw denial spikes while pool appeared underutilized; the “laggy” telemetry made the controller think they were still consuming burst tokens.
- Customer-visible effect: burst requests denied unexpectedly; some clients retried aggressively and amplified load.
- Root cause (likely): our admission controller relied on near-real-time usage signals; when signals were delayed, the smoothing window + token bucket refill behavior effectively penalized active tenants.
- Fix during beta: hotfix added smoothing + guard conditions + conservative fallback when telemetry freshness is below threshold.
- Ref: GitHub:redwood/pr-23944

B) EU-West burst admission timeouts (controller dependency chain)
- Symptom: admission calls intermittently timed out in eu-west under high request rate; routing fell back but increased gateway error rates briefly.
- Root cause: admission controller p99 latency spiked due to dependency (policy evaluation + fleet telemetry query) not being bounded; threadpool saturation.
- Mitigation: enable circuit breaker + temporarily tighten per-tenant caps + disable burst in eu-west allow list until patched.
- Ref: Jira:SUP-28579

C) Billing: burst usage double-counted on retry paths
- Symptom: customer asked why burst line item didn’t match dashboard usage; we found ~1.8–2.3% overcount in a specific retry pattern.
- Root cause: metering event emitted pre-idempotency; retried requests emitted duplicate usage increments.
- Fix: idempotency keys + reconciliation checks.
- Ref: GitHub:redwood/pr-24108, Jira:SUP-28488

D) Noisy neighbor scare (tail latency increase for a non-burst tenant)
- Symptom: p95/p99 latency increase for a non-burst tenant correlated with another tenant’s burst window.
- What we learned: even if pool isolation is “logical,” shared control-plane/routing and some shared fleet components can cause tail amplification under saturation.
- Mitigation used: reduced global cap + increased cooldown for bursting tenant + forced routing fallback for that tenant for 30 min.
- Ref: Jira:SUP-28455

E) Support confusion: 429s when burst cap reached (expected but poorly messaged)
- We returned 429/overload when both baseline+burst caps were reached; some customers expected queuing instead.
- For GA we need crisp guidance: recommended retry/backoff strategy and what will/won’t queue.
- Ref: Jira:SUP-28503


Guardrails that were too strict (beta feedback)
1) Cooldown windows were overly punitive for spiky-but-legit workloads
- Current behavior: once a tenant hits max burst ratio, cooldown kicks in even if they drop below quickly.
- Effect: denial bursts after short spikes; customers perceive “flaky burst.”
- Proposal: cooldown should be based on sustained saturation (e.g., N seconds over threshold) or make cooldown proportional to burst duration, not a flat window.

2) Max burst ratio default (x1.5) was too conservative for “weekly batch” customers
- Several beta tenants expected x2–x3 for short windows (10–30 min) and were willing to pay.
- But we must pair higher ratios with stronger fairness + fleet safety + SLO gates.

3) Admission determinism vs latency tradeoff
- We enforced determinism by evaluating multiple signals synchronously; in some cases this added controller latency and increased timeouts.
- Suggestion: determinism requirement should be: “same inputs => same decision” but inputs can be snapshot-based (cached) within a bounded TTL.


Guardrails that were too lax (or missing)
1) Retry amplification guard
- When customers got denials, some clients retried immediately with high concurrency.
- We need an explicit “retry-after” / backoff hint tied to denial reason codes, and optionally server-side throttling at gateway.

2) Alerting gaps for “telemetry freshness” and “admission controller dependency health”
- We lacked a strong, paging-quality indicator for telemetry freshness degradation (it showed in graphs but didn’t page until customers complained).
- Admission controller dependency p99 did not have an error-budget aligned alert.

3) Fairness under mixed priority tiers needs stronger anti-starvation proofs
- We introduced priority tiers late in beta; we didn’t run enough multi-tenant stress tests for gold/silver combos.
- Need explicit starvation bound (e.g., min guaranteed tokens) and test harness enforcement.

4) Config propagation / console staleness
- One customer ticket: burst limit updated but Console showed stale value for too long.
- Root cause likely caching/prop delay; not a safety issue but it undermines trust.
- Ref: Jira:SUP-28541


Key metrics from beta (rough, from dashboards)
- Total burst attempts: dominated by 2 tenants with scheduled batch jobs.
- Denial rate: ranged 0–18% by tenant depending on time-of-day; spikes correlated with telemetry lag and cooldown behavior.
- Admission controller p99: stable in us-east; spiky in eu-west (timeouts).
- Neighbor impact: one validated event (SUP-28455) where non-burst p99 regressed ~12–18% for ~20 minutes before mitigation.
- Billing reconciliation: identified retry double-counting bug; now fixed but we need ongoing automated checks.

Open questions (need decision)
1) Queue vs immediate denial for pool saturation
- Today we mostly deny quickly; some customers want queuing.
- SRE view: queuing can hide fleet pain and increase tail; if we add queueing it must be bounded with clear max wait + cancellation and must not break determinism.

2) How much “degraded mode” behavior is acceptable?
- If telemetry is stale, do we:
  a) deny all burst? (safe but may be overly strict)
  b) allow small burst with conservative cap? (better UX)
  c) continue as-is with risk? (not acceptable)
- Tentative: (b) with tight global caps + explicit surfaced state.

3) GA scope: region/hardware allow list
- Recommendation: GA in us-east H100 first; add eu-west after EU-West latency fix + game day rerun.


Appendix: customer support learnings (from tickets)
- SUP-28411 (unexpected denials): customers interpret “burst enabled” as “guaranteed extra capacity.” We must emphasize “best-effort within guardrails,” with clear reasons.
- SUP-28503 (429s): customers need explicit guidance on client-side backoff, concurrency control, and how to detect “you are at cap.”
- SUP-28488 (billing question): customers will audit the burst line item against their own job schedule; we need accurate exports + transparent calculation.
