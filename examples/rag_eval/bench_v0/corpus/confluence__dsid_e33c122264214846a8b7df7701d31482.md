Dedicated Throughput SLO (Formal Definition)

## Purpose
This page defines the canonical Service Level Objective (SLO) for **Dedicated throughput** at Redwood Inference, including:
- SLI and metric semantics (tokens/sec as canonical; when requests/sec is used)
- Measurement windows and aggregation
- Exclusions and “not counted” conditions
- Burn-rate alerting and error budget policy
- Reporting requirements and ownership

This is the formal reference used by SRE, Observability, Infra/Fleet, and Product when implementing dashboards, alerts, incident classification, and customer-facing reporting.

## Scope
### Applies to
- **Dedicated deployments** (hosted in Redwood-managed infrastructure) with a **committed throughput floor** recorded in control plane configuration.
- Deployments where a customer has:
  - A pinned **model family + major version** (e.g., `llama-3.x`, `mistral-large.x`) and
  - A configured **max context length** within the supported range for the purchased hardware shape.

### Does not apply to
- Shared/Hosted API (non-dedicated) traffic
- Private (VPC/on-prem) deployments (separate SLOs)
- “Best-effort” Dedicated trials without a committed throughput floor

## Definitions
- **Deployment**: A Dedicated inference endpoint identified by `deployment_id` and bound to a region and hardware pool.
- **Route**: A logical API capability within a deployment (e.g., `chat.completions`, `completions`, `embeddings`). For this SLO, throughput is evaluated at the **deployment level**, with route breakdowns used for diagnostics.
- **Committed Throughput Floor (CTF)**: The minimum throughput Redwood commits to sustain for a deployment under defined conditions. Expressed as **tokens/sec** (canonical) or **requests/sec** (only for non-tokenized routes; see below).
- **Observed Throughput**: Throughput computed from Redwood-side metering and timing, scoped to eligible requests.
- **SLO Window**: The evaluation interval used to determine compliance for a point in time (5 minutes).
- **Error Budget**: The allowed fraction of SLO windows that can be below the committed floor over the compliance period.

## Canonical SLI and SLO
### Canonical throughput metric (SLI)
**SLI name:** Dedicated Throughput Floor Attainment

**SLI definition (per deployment):**
For each `deployment_id`, compute the fraction of eligible 5-minute windows in which the **observed output token rate** meets or exceeds the committed floor.

A 5-minute window is considered **good** if:
- `observed_tps(deployment_id, window) >= committed_tps(deployment_id) * 1.00`

A window is **bad** if:
- `observed_tps(deployment_id, window) < committed_tps(deployment_id) * 1.00`

**Why tokens/sec:** Token throughput is the most stable cross-model, cross-request indicator for generation routes and aligns with customer-perceived capacity for streaming/chat.

### SLO target
**SLO (per deployment):**
- **99.5%** of eligible 5-minute windows over a rolling **30-day** period must be good.

This implies an error budget of **0.5%** of windows per 30 days.

### Requests/sec as a secondary metric
Requests/sec may be used for:
- Non-generative routes where token accounting is not meaningful (rare in Dedicated scope), or
- As an internal diagnostic view (e.g., “QPS at steady state”) alongside tokens/sec.

If requests/sec is used, it is explicitly labeled as a **non-canonical** SLI and must not be substituted for compliance without SRE sign-off.

## Measurement semantics
### Windowing and aggregation
- **Evaluation window:** 5 minutes (tumbling windows)
- **Compliance period:** 30 days (rolling)
- **Aggregation level:** `deployment_id` (primary); `deployment_id + route + model` (debug)

### Observed throughput calculation (tokens/sec)
For each deployment and 5-minute window:
- `observed_tps = eligible_output_tokens / eligible_generation_wall_time_seconds`

Where:
- **eligible_output_tokens** counts output tokens generated for eligible requests whose generation occurs within the window.
- **eligible_generation_wall_time_seconds** is the sum of per-request generation durations for eligible requests, measured at Redwood’s gateway/runtime boundary.

Notes:
- We use **wall time spent generating** rather than total window duration to avoid penalizing low-utilization periods.
- This SLO measures our ability to serve when the customer is asking for capacity, not whether the customer sends enough traffic to saturate the deployment.

### Eligibility rules (what counts as “eligible” load)
A request is eligible if all of the following are true:
1) The request is accepted for processing (HTTP 2xx; not rejected by admission control).
2) The request targets a model and context configuration within the deployment’s contracted support bounds.
3) The request is not customer-throttled (see exclusions).
4) The request is not in a cold/warm-up exclusion window after provisioning or scale events (see exclusions).

### Streaming token accounting
For streaming responses:
- Output tokens are counted when emitted by the runtime, attributed to the originating request.
- We de-duplicate token counters to avoid double counting caused by stream chunk retries.

Operational note: streaming accounting semantics were corrected in **PR-19422** (“Fix throughput metric double counting streaming”). Any backfills or comparisons across that boundary must annotate the change.

### Batching and caching
- Continuous batching and prefix/KV caching are treated as implementation details and are inherently included in observed throughput.
- Customers may see throughput increases due to caching; this does not change the committed floor.
- SLO compliance is evaluated on Redwood-side metered tokens/time, not customer-side client timers.

## Exclusions (non-counted windows/requests)
Exclusions exist to avoid situations where the throughput floor is not meaningful, is externally constrained, or is not under Redwood’s control.

### Request-level exclusions
The following requests are excluded from both numerator and denominator:
- **Customer-side throttling or pacing** that prevents the deployment from being demand-saturated while measuring throughput (e.g., the customer sends traffic well below the committed floor and attempts to interpret low throughput as non-compliance).
- **Client disconnects / cancellations** before generation completes (where Redwood terminates work early).
- **Invalid requests** or requests that violate documented limits (e.g., context > supported max for the purchased shape, unsupported model version).
- **Customer-enforced caps** that materially reduce achievable throughput, including:
  - Very low `max_tokens` combined with high request overhead patterns (evaluated case-by-case; see “edge-case review” below)
  - Customer-set concurrency limits below recommended saturation settings

### Window-level exclusions
The following 5-minute windows may be excluded (removed from total window count) if conditions are met and are annotated in reporting:
- **Provisioning warm-up:** First **15 minutes** after a deployment becomes active.
- **Scale event warm-up:** First **10 minutes** after an approved scale-up/scale-down event that changes active capacity.
- **Planned maintenance:** Approved maintenance windows with customer notice, tracked via change calendar and incident tooling.

### Edge-case review (must be documented)
If a customer workload has atypical characteristics (e.g., extremely long contexts, highly skewed token distributions, tool-calling patterns with minimal output), the account team must request an SRE review to confirm:
- The correct committed floor unit (tokens/sec vs alternative) and
- The correct eligibility and saturation guidance.

## Saturation requirement (practical interpretation)
This SLO is designed to answer: “When the customer is **trying** to use their Dedicated capacity, do we deliver the committed throughput?”

Therefore, compliance reporting only considers periods where the deployment is **meaningfully loaded**.

We define “meaningfully loaded” for a window as:
- `eligible_generation_wall_time_seconds >= 120` seconds within the 5-minute window

If a window does not meet the load threshold, it is treated as **insufficient demand** and excluded from compliance (removed from total). This prevents underutilized deployments from producing misleading ‘low throughput’ windows.

## Metrics (source of truth)
### Required metric labels
All metrics used for SLO evaluation must include:
- `deployment_id`
- `region`
- `tier` (Standard/Plus/Premium)
- `model` (resolved model id)
- `route`

### Canonical metrics (Prometheus-style names)
The evaluator uses the following canonical signals (names reflect our current convention; see the metric registry for exact schemas):
- `ri_dedicated_output_tokens_total{deployment_id,...}`
- `ri_dedicated_generation_seconds_total{deployment_id,...}`
- `ri_dedicated_requests_accepted_total{deployment_id,...}`
- `ri_dedicated_requests_rejected_total{deployment_id,reason,...}`
- `ri_dedicated_streaming_chunks_total{deployment_id,...}` (diagnostic)

Control plane configuration source:
- `ri_dedicated_committed_tps{deployment_id}` (exported from deployment config service)

## SLO evaluation logic (reference)
For each deployment_id and 5-minute window:
1) Compute eligible tokens and eligible generation seconds.
2) If `eligible_generation_seconds < 120`, mark window as **excluded_insufficient_demand**.
3) Else compute `observed_tps = tokens / seconds`.
4) Window is good if `observed_tps >= committed_tps`.

The rolling SLO is:
- `good_windows / eligible_windows >= 0.995` over 30 days.

## Burn-rate alerting
We alert on rapid consumption of the 30-day error budget using burn-rate alerting.

### Parameters
- SLO target: 99.5% (error budget = 0.5%)
- Window size for good/bad decisions: 5 minutes
- Alerts evaluated per `deployment_id` (not fleet-wide)

### Paging vs ticketing
**Paging alerts (customer-impacting / requires oncall response):**
- **Fast burn:** burn rate >= **14x** over **1h** AND burn rate >= **6x** over **6h**
  - Interpretation: likely to exhaust error budget within ~2.1 days if sustained.

**Ticket alerts (investigate during business hours):**
- **Slow burn:** burn rate >= **3x** over **24h**
  - Interpretation: will exhaust error budget within ~10 days if sustained.

### Burn-rate computation
For a given lookback window `T`:
- `error_rate_T = bad_windows_T / eligible_windows_T`
- `burn_rate_T = error_rate_T / 0.005`

Notes:
- Excluded windows (maintenance, warm-up, insufficient demand) are removed from `eligible_windows_T`.
- If `eligible_windows_T` is below a minimum sample threshold (default: >= 6 eligible windows), burn-rate alerts are suppressed to reduce noise.

### Alert routing
- Page: `#eng-oncall` and oncall rotation “SRE - Inference Platform”
- Auto-attach runbook link: “Dedicated Throughput SLO Escalation”
- Auto-include fields: `deployment_id`, `customer_name` (if available), `tier`, `region`, current `observed_tps`, `committed_tps`, top rejection reasons, saturation signals

## Error budget policy (operational gates)
Error budget policy is evaluated per deployment_id.

### Policy thresholds (rolling 30-day)
- **Healthy:** >= 70% error budget remaining
  - Normal change velocity.
- **Watch:** 30–70% error budget remaining
  - Requires SRE review for changes that can impact serving performance (runtime upgrades, kernel toggles, routing policy changes).
- **At risk:** < 30% error budget remaining
  - Freeze performance-risky changes for affected tier/pool unless explicitly approved by SRE lead (Rafael Mendes) and Infra lead (Irene Choi).
- **Exhausted:** SLO below target
  - Treat as an incident for Dedicated customers; engage CS per escalation playbook; credits/remedies handled per commercial terms.

### What counts as “performance-risky” change
Examples:
- Runtime version upgrade, quantization profile change, kernel selection changes
- Routing/admission control changes affecting Dedicated pools
- Fleet capacity rebalancing that changes effective headroom

## Reporting
### Internal reporting (required)
- Daily SLO summary per deployment for SRE and Dedicated Product
- Weekly rollup by tier and region for capacity planning
- Incident linkage: SLO violations must link to incident tickets and postmortems where applicable

### Customer-facing reporting (when applicable)
- Provide a monthly compliance summary upon request for Plus/Premium tiers.
- Reports must:
  - State the committed throughput floor
  - State the achieved compliance (% good windows)
  - List excluded windows with category and timestamps (maintenance, warm-up)
  - Include a short explanation of the “insufficient demand” exclusion

## Appendix: Examples
### Example 1: Healthy compliance
- Committed: 2,500 tokens/sec
- Over 30 days: 8,000 eligible windows
- Bad windows: 20
- Error rate: 0.25%
- Compliance: 99.75% (meets 99.5%)

### Example 2: Underutilized deployment
- Customer sends sporadic traffic; most 5-minute windows have < 120 seconds of generation time.
- Those windows are excluded as insufficient demand.
- Compliance is computed only on windows where the deployment is meaningfully loaded.

### Example 3: Customer asks for unsupported context
- Customer sends requests with context > supported max for purchased shape.
- These requests are excluded; account team must re-scope the commitment or upsell shape/tier.
