# Sup 18510 Streaming Throughput Metric Mismatch

Source type: jira
Document ID: dsid_a5a5235d973e421c870e11ceea4887ed
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Streaming throughput metric mismatch (customer dashboard vs Redwood export) — suspected token accounting double-count in streaming

Customer (Vidorama) reports inconsistent throughput numbers for their Dedicated deployment when using streaming responses.

Issue summary
- Vidorama is comparing:
  1) Redwood Console throughput panel (tokens/sec) for their Dedicated deployment, and
  2) Their internal measurement based on streamed tokens (client-side) + Redwood usage export.
- They see Redwood Console tokens/sec ~1.8–2.1x higher than their client-side computed throughput for the same time windows.
- Discrepancy appears only for streaming (non-streaming requests match within ~5–10%).

Impact
- Customer confusion + concern about Dedicated throughput SLO reporting accuracy.
- Potential false positives/negatives in SLO compliance if the canonical throughput metric is inflated for streaming.

Customer-provided details
- Workload: chat completions, streaming enabled (server-sent events).
- Typical request: prompt 1–2k tokens, completion 150–400 tokens, 20–60 concurrent streams.
- Sampling window: 5-min rolling, comparing against their Prometheus counters.

Redwood initial checks (Support)
- Confirmed customer is hitting the correct deployment_id and region (us-east).
- Confirmed they are not mixing model variants in the comparison window (pinned to llama-3.1-70b-instruct; occasional mixtral bursts were excluded by customer).
- Confirmed they are not counting input tokens in their client throughput metric (they only count generated tokens).

Steps to reproduce (internal)
1) Run load test against a Dedicated deployment with streaming enabled.
2) Compare:
   - streaming token counter used by throughput dashboard (server-side), vs
   - sum(completion_tokens) from request final usage events (server-side) and/or reconstructed deltas.
3) Observe that for streaming requests, the dashboard-aligned metric increases faster than final usage totals.

Investigation notes (SRE/Obs)
- Hypothesis A: double-counting streamed completion tokens (both per-chunk deltas and final usage event contribute to the same counter).
- Hypothesis B: retries/resumes on stream reconnects cause duplicated deltas.
- Hypothesis C: mixed attribution labels (route vs deployment) cause aggregation overlap.

Findings
- We can reproduce consistent ~2x inflation when streaming is enabled and client consumes the full stream.
- Root cause identified in telemetry emission path:
  - For streaming responses, we emit per-chunk token deltas (used for near-real-time throughput).
  - At stream termination, a final usage event is emitted.
  - In the current pipeline, the dashboard throughput counter was incremented by BOTH the per-chunk deltas and the final usage event for the same request.
- This does NOT affect billing totals (billing uses finalized usage only). It affects the real-time throughput metric used in dashboards and SLO evaluation.

Workaround (until fix rollout)
- For customer reporting, use finalized completion token metrics (request end) for streaming workloads:
  - completion_tokens_total (finalized) / wall time
- Avoid using real-time streaming throughput panel for contractual/SLO comparisons until patched.

Next actions
- Engineering to patch streaming accounting to ensure final usage does not also increment the streaming delta counter (or to separate metrics cleanly).
- Backfill: no backfill planned; we will annotate the dashboard for the affected time range and document the known issue.
- Customer comms: provide written explanation that billing is unaffected; commit to patch timeline and a short validation report post-deploy.
