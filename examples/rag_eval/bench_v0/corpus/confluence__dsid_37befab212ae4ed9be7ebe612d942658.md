Billing Meter Spec — Hosted API Tokens & Requests

## 1. Purpose
This page defines the **formal billing meters** for Redwood Inference’s **Hosted API** (including OpenAI-compatible endpoints) and the rules used to compute:
- **Request counts**
- **Prompt (input) tokens**
- **Completion (output) tokens**
- **Billed tokens** (the chargeable unit for usage-based pricing)

It also specifies:
- Streaming semantics and disconnect behavior
- Retries, idempotency, and deduplication requirements (to prevent double-counting)
- Tool/function calling token attribution rules
- Rounding rules
- How meters map to the **billing ledger**, **customer Usage API**, and **invoices**

This spec is used by Engineering (telemetry + billing hooks), Finance/RevRec (reconciliation + close), RevOps/Product (pricing experiments), and Legal (usage definitions in contracts).

---

## 2. Scope
### 2.1 Included surfaces
This spec applies to Hosted API endpoints that return model outputs and therefore generate token usage:
- Chat / Responses-style generation endpoints
- Text completion endpoints
- Rerank endpoints **only if** they are billed per token (currently: billed per request; see notes)
- Embeddings endpoints are covered by a separate meter spec (batch semantics differ)

### 2.2 Exclusions
- Dedicated capacity GPU-seconds meters (separate spec)
- Private deployments (metering may be customer-operated; contracts may reference this spec as guidance only)

---

## 3. Principles (must-haves)
1. **Single source of truth for invoicing:** The billing ledger is the authoritative source for invoice totals. Telemetry events are authoritative for debugging and reconciliation.
2. **Idempotent billing:** Retries/fallbacks must not produce double-charges when the customer provided a stable idempotency key.
3. **Customer-observable semantics:** Usage API must explain billed vs raw tokens and any cache attribution.
4. **Consistency across endpoints:** OpenAI-compatible endpoints must align token counting semantics with the canonical Redwood endpoints (modulo model/tokenizer differences).
5. **Change control:** Any meter logic change requires shadow compare and Finance approval before impacting invoices.

---

## 4. Definitions
### 4.1 Core terms
- **Request:** A single client call to a Hosted API endpoint.
- **Attempt:** A server-side execution attempt for a request (may include fallback routing or server-side retry).
- **Prompt tokens (input_tokens):** Tokens attributable to the serialized input to the model for an attempt.
- **Completion tokens (output_tokens):** Tokens generated by the model for an attempt.
- **Raw tokens:** input_tokens + output_tokens as observed for an attempt.
- **Billed tokens:** Token quantity used for charging. May differ from raw tokens when cache hits or policy-based adjustments apply.
- **Cache hit:** A case where a prefix/KV cache or response cache results in reduced compute for the request. Depending on pricing policy, cache-hit tokens may be billed differently (see §7).

### 4.2 Identifiers
- **account_id:** Billing account (maps to contract / invoice recipient).
- **workspace_id / project_id:** Sub-account segmentation (if enabled).
- **request_id:** Redwood-generated ID for a request.
- **idempotency_key:** Customer-provided string for deduplication of retries.
- **billing_dedupe_key:** Canonical key used in billing events and ledger to enforce idempotency.

### 4.3 “Chargeable event time”
All Hosted API meters roll up by **UTC usage date**, based on `metered_at` (see §10). Export/UI may render in customer timezone, but the ledger uses UTC for close.

## 5. Meter catalog (Hosted API)
The following meters are produced for every billable request.

### 5.1 Meter names and units
| Meter | Field name (internal) | Unit | Notes |
|---|---|---|---|
| Requests | request_count | count | Counted when request is billable per §6 |
| Prompt tokens | prompt_tokens | tokens | Integer tokens for serialized input |
| Completion tokens | completion_tokens | tokens | Integer tokens generated |
| Raw tokens | raw_tokens | tokens | prompt_tokens + completion_tokens |
| Billed tokens | billed_tokens | tokens | Primary pricing unit for Hosted API generation |

### 5.2 Required dimensions
Every usage record MUST include the following dimensions (nulls are not allowed unless explicitly marked optional):
- `account_id` (required)
- `workspace_id` (optional)
- `project_id` (optional)
- `api_key_id` (required; may be hashed in customer exports)
- `endpoint` (required; canonical string, e.g., `chat.completions`, `responses`, `completions`)
- `model` (required; resolved to the **billed model** after routing/fallback)
- `model_variant` (optional; internal)
- `region` (required)
- `deployment_mode = hosted` (required)
- `pricing_plan_id` (required; ties to packaging/experiment cohorts)
- `currency` (required; pricing table lookup)
- `meter_version` (required; see §11)
- `metered_at` (required; UTC timestamp)

Optional but strongly recommended:
- `customer_request_tag` (optional; customer-provided metadata)
- `experiment_cohort` (optional; when experiment flagging is active)

## 6. Request counting rules (request_count)
### 6.1 When a request is counted
`request_count = 1` if and only if all conditions below are true:
1. The request is accepted by Redwood (passes auth + basic validation).
2. The request is not deduplicated by idempotency (see §8).
3. The request is **billable** based on response category:
   - **2xx** responses: billable.
   - **4xx** responses: not billable, except where explicitly stated (none currently).
   - **5xx** responses: billable only if tokens were produced and streamed/returned (see §6.3 and §7.4).

### 6.2 Partial success
If a request returns partial output (e.g., streaming interrupted after some tokens), it is still counted as a request if it is billable per §7.4.

### 6.3 Streaming and request_count
For streaming endpoints:
- `request_count` is emitted once per **deduped request**.
- If the client disconnects mid-stream, the request still counts if billing conditions are met (see §7.4).

## 7. Token metering rules
### 7.1 Canonical tokenization
Token counting uses the tokenizer associated with the **resolved billed model**.
- If routing selects a different compatible model variant than requested (fallback), the billed tokens MUST reflect the billed model’s tokenizer.
- Usage API must return both `requested_model` and `billed_model` when they differ.

### 7.2 Prompt tokens (prompt_tokens)
Prompt tokens represent the total tokens in the model input for the billed attempt, including:
- System/developer/user messages (for chat)
- Tool schema/function definitions provided by the client (for tool calling)
- Any platform-inserted control tokens required by the serving runtime for the endpoint format

Rules:
- Count prompt tokens once for the billed attempt.
- If internal reformatting occurs (e.g., OpenAI-compatible -> canonical), token counting is performed after canonicalization.

### 7.3 Completion tokens (completion_tokens)
Completion tokens represent the total tokens generated and delivered by the model attempt.
- For streaming, tokens are counted as generated, and the final total is reported at end-of-stream.
- If the stream terminates early (disconnect), completion tokens include only the tokens produced up to termination.

### 7.4 Billed tokens (billed_tokens)
`billed_tokens` is the default basis for charging and is computed as:

**billed_tokens = billed_prompt_tokens + billed_completion_tokens**

Unless otherwise specified by pricing policy, the default is:
- `billed_prompt_tokens = prompt_tokens`
- `billed_completion_tokens = completion_tokens`

Billing behavior by outcome:
- **Successful 2xx:** bill billed_tokens per above.
- **Client disconnect during streaming:** bill tokens actually generated and sent prior to disconnect (no future/estimated tokens). Request is billable if at least one token was streamed OR if prompt processing completed and policy bills prompt tokens (current policy: bill prompt tokens if the model began execution; see §7.6).
- **Server error after tokens produced:** bill tokens produced (prompt + completion as applicable). Must not bill tokens for attempts that never started execution.

### 7.5 Cache-hit attribution and billed tokens
Hosted API may apply caching at different layers:
- Prefix/KV cache (compute reduction, same semantic output generation)
- Response cache (replay exact output)

Metering requirements:
- Record `cache_hit = true/false` and `cache_type` (optional: `kv_prefix`, `response`, `other`).
- Record `raw_tokens` as normal.
- Record `billed_tokens` according to active pricing plan policy.

Current billing policy (default unless overridden in plan):
- Cache hits do **not** change `billed_tokens` (billed remains aligned to customer-visible usage). Cache attribution is used for margin analysis and pricing experiments.

If a future plan bills cache hits differently, that plan must:
- Explicitly define `cache_billing_policy` (e.g., `full`, `discounted`, `free_prompt`, etc.)
- Expose that policy in contract/order form language as required by Legal
- Be introduced only with meter_version bump and shadow compare (§11)

### 7.6 When prompt tokens are billable for interrupted requests
For streaming requests that fail or disconnect:
- If **model execution began** (i.e., prompt was tokenized and submitted to the runtime), then prompt_tokens are billable.
- If the request failed before runtime submission (auth failure, validation error, quota rejection), prompt_tokens are not billable.

Engineering signal:
- `execution_started = true/false` must be present on usage events.

## 8. Retries, fallback routing, and idempotency (no double-counting)
### 8.1 Idempotency key behavior
If a client provides `Idempotency-Key`, Redwood must ensure:
- Only one billable usage record is emitted per unique `(account_id, endpoint, idempotency_key)` within the dedupe window.
- Additional retries return the same `request_id`/result where possible, or a consistent `original_request_id` pointer.

### 8.2 Billing dedupe key
`billing_dedupe_key` MUST be computed as:
- Preferred: a stable internal request UUID generated on first acceptance and persisted across retries.
- Acceptable: hash of `(account_id, idempotency_key, endpoint)` if internal request UUID is unavailable.

Dedupe window:
- Minimum: 24 hours
- Target: 7 days (to cover client retry storms and delayed networks)

### 8.3 Server-side retries and fallback routing
Server-side retries/fallbacks may create multiple attempts. Metering rules:
- Usage events may record multiple **attempt-level** diagnostics.
- Billing meters MUST roll up to exactly one **billed request** per dedupe key.
- The billed model is the model that produced the returned output.
- If a first attempt produced tokens but was not returned to the customer (e.g., internal failure before any bytes were sent), those tokens are not billable.

### 8.4 Required fields for retry safety
Usage events MUST include:
- `attempt_id`
- `attempt_index` (0,1,2,...)
- `original_request_id` (if retry)
- `billing_dedupe_key`
- `deduped = true/false`
- `retry_reason` (optional)

## 9. Tool / function calling token attribution
### 9.1 What counts as prompt tokens
The following are part of prompt tokenization:
- Tool/function schema definitions provided with the request
- Prior tool call results included in the message history
- Any structured output schema definitions (JSON schema, response format) supplied by the client

### 9.2 What counts as completion tokens
The following are part of completion tokens:
- The model’s emitted tool call arguments (even if the client later rejects them)
- The model’s structured JSON output tokens

### 9.3 Multiple tool calls in one response
If the model emits multiple tool calls:
- All emitted tokens count toward `completion_tokens`.
- No special discounts/adjustments are applied unless a pricing plan explicitly states otherwise.

### 9.4 Tool execution is not a billing event
Execution time or cost of downstream tools is not billed via token meters (it may be billed via separate product charges in the future).

## 10. Rounding and numeric invariants
### 10.1 Token integer requirement
All token meters MUST be whole integers (no fractional tokens).

### 10.2 Rounding
- Tokenizers yield integer counts; no rounding should be applied.
- If an upstream component provides non-integer token counts (bug), the meter pipeline must:
  - Reject the event from billing aggregation (do not invoice)
  - Emit an alert (meter data-quality SLO breach)
  - Preserve the raw event in quarantine for investigation

### 10.3 Invariants
For each billed request record:
- `raw_tokens = prompt_tokens + completion_tokens`
- `billed_tokens >= 0`
- `prompt_tokens >= 0`, `completion_tokens >= 0`
- `billed_tokens` must be explainable by policy; if `billed_tokens != raw_tokens`, a non-null `billing_adjustment_reason` must be set.

## 11. Meter versions and change control
### 11.1 meter_version
All Hosted API usage records must carry `meter_version`.
- Current target: `hosted_api_tokens_v1`
- Any behavioral change that can impact billed quantities requires a new version (e.g., `v2`).

### 11.2 Shadow billing compare
Before enabling a new meter_version for invoicing:
1. Run **shadow billing compare** for at least 7 consecutive days across representative traffic.
2. Produce an impact report by account/model/region/endpoint (P50/P95 deltas; outlier accounts).
3. Obtain approvals:
   - Finance/RevRec (Emily Stone)
   - Telemetry Eng owner (Logan Wright)
   - Product owner for Hosted API pricing (Nikhil Sharma)
   - Legal if contract language is impacted (Michael Grant)

### 11.3 Backfills
Backfills that would change historical billed usage require explicit Finance approval and must follow the reconciliation playbook.

## 12. Mapping to Usage API (customer-facing)
### 12.1 Usage API field requirements
For transparency and dispute reduction, the customer Usage API must expose (at minimum):
- `request_count`
- `prompt_tokens`
- `completion_tokens`
- `billed_tokens`
- `model` (requested)
- `billed_model` (if different)
- `endpoint`
- `region`
- `metered_at` (UTC) and `usage_date_utc`
- `cache_hit` (boolean) and `cache_type` (if available)
- `idempotency_key_present` (boolean; do not expose raw key)
- `meter_version`

### 12.2 Customer-visible definitions
Usage API documentation must use the same definitions as this spec, especially:
- “Billed tokens vs raw tokens”
- How streaming disconnects are handled
- How retries with idempotency keys are handled

## 13. Mapping to invoices and billing ledger
### 13.1 Billing ledger aggregation
Invoices are produced from the billing ledger aggregated by:
- account_id
- billing period
- pricing plan / contract terms
- model and endpoint (if priced differently)

Ledger line item computation:
- `line_item_quantity = SUM(billed_tokens)` (for token-priced products)
- `line_item_quantity = SUM(request_count)` (for request-priced products, if any)

### 13.2 Invoice display conventions
Invoice line items should display:
- Product name (Hosted API)
- Meter name (Billed tokens or Requests)
- Unit price
- Quantity
- Total

If customers are on a plan with cache-specific billing, invoices must include language to avoid confusion (Legal-approved wording required).

## 14. Examples (normative)
### Example A — Standard non-streaming chat completion
Input: 120 prompt tokens; output: 80 completion tokens.
- request_count = 1
- prompt_tokens = 120
- completion_tokens = 80
- raw_tokens = 200
- billed_tokens = 200

### Example B — Streaming completion, client disconnect after 20 tokens
Prompt tokens: 150. Client disconnects after receiving 20 completion tokens. Execution started.
- request_count = 1
- prompt_tokens = 150
- completion_tokens = 20
- billed_tokens = 170

### Example C — Server-side retry with idempotency key (no double-count)
Client sends request with Idempotency-Key. First attempt fails before any bytes sent; second attempt succeeds.
- Only one billed request record emitted.
- request_count = 1
- prompt_tokens/completion_tokens reflect the successful attempt.
- No billed tokens from first attempt if nothing was returned and execution did not produce delivered tokens.

### Example D — Fallback routing changes billed model
Requested model `rw-llama-70b`, billed model `rw-llama-70b-fast` due to capacity.
- Tokenization uses billed model’s tokenizer.
- Usage API returns both requested and billed model fields.

### Example E — Tool calling
Tool schema adds 300 prompt tokens; model emits tool call args 50 tokens.
- prompt_tokens includes tool schema.
- completion_tokens includes tool call args.
