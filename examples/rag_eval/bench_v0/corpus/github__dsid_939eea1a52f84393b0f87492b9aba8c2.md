# Pr 24108 Fix Billing Double Counting On Burst Retries

Source type: github
Document ID: dsid_939eea1a52f84393b0f87492b9aba8c2
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Fix billing double-counting on Dedicated burst retries (idempotency + reconciliation)

Problem: we are double-counting burst usage when the same customer request is retried (client retries, gateway retries after 5xx/timeout, and some 429 backoff paths). In beta, this showed up as duplicate burst usage line items for the same request window and was flagged in SUP-28488 and during reconciliation sampling. Root cause: the burst metering emitter currently treats each attempt as billable because we do not have a stable request-scoped idempotency key carried end-to-end into the metering aggregation stage; request_id/trace_id are not consistently preserved across retry code paths. Fix: introduce a request-scoped idempotency key that is generated at the gateway if missing, propagated to the data plane and metering emitter, and used to dedupe burst billable events before exporting to billing. Also add a reconciliation guard that detects duplicates that slip through (e.g., telemetry lag / partial outages) and blocks export with an alert. Details: 1) API/gateway: accept and forward X-Redwood-Idempotency-Key; if absent, generate a ULID per incoming request and reuse it across internal retry attempts. 2) Metering: include idempotency_key on burst usage events; dedupe in the burst_usage_aggregator on (tenant_id, region, hw_class, pool_id, idempotency_key) with a bounded TTL and hashed storage to avoid high-cardinality raw keys. 3) Billing export: export a deduped_usage_count and dropped_duplicate_attempts metric; add a hard reconciliation check that fails the export batch if duplicate keys exceed a small threshold (defaults to 0.01%) and pages on-call for billing pipeline. 4) Tests: add integration test for retry paths to ensure only one billable burst event per idempotency key; add unit tests for dedupe window behavior and hash collisions (expected negligible; we still include tenant/pool scoping). Rollout: behind feature flag billing.burst_idempotency_dedupe_enabled (default off). Plan is to enable in staging immediately, then 10% of Dedicated burst tenants in us-east for 24h, then ramp. Risk: incorrect dedupe could under-bill if keys are reused incorrectly; mitigated by scoping on tenant+pool and generating server-side keys when absent. Note: this PR does not change customer-visible request semantics; it only makes billing/metering idempotent across retries.
