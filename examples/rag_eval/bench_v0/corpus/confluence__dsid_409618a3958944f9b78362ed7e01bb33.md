Dedicated Throughput SLO Escalation Playbook

## Purpose
This playbook defines the end-to-end process for handling Dedicated **Throughput SLO** misses or suspected misses. It is designed for Customer Success (CS) and Support to:
- Triage quickly and consistently.
- Engage the correct on-call teams (SRE / Infra / Observability) at the right time.
- Communicate clearly with customers, including status updates and written summaries.
- Route service-credit / compensation requests through the correct approval path.

This page is **restricted** because it includes internal paging triggers and credit approval guidance.

---

## What is the Dedicated Throughput SLO (customer-facing summary)
Dedicated Throughput SLO guarantees a **minimum sustained throughput floor** for an eligible Dedicated deployment under defined conditions.

**High-level definition (internal summary):**
- Metric: throughput measured as **billable output tokens per second**, computed per deployment, aggregated over a rolling window.
- Scope: Dedicated deployments only (not Hosted API, not eval/sandbox deployments).
- Eligibility: model/version and configuration must be within the customer’s contracted Dedicated configuration (e.g., pinned model family/version, supported max context, supported shape).

**Common non-eligibility / exclusions (not exhaustive):**
- Customer-side throttling or client concurrency limits causing underutilization.
- Customer workload violating sizing assumptions (e.g., contexts above configured limit, unsupported parameter combinations).
- Warm-up periods after scale-from-zero or cold start (tier-dependent; see “Warm pool / startup” below).
- Incidents due to customer networking/VPC misconfig in Private connectivity paths (if applicable).

When in doubt, treat as potentially eligible, start the triage, and engage SRE.

## Roles and ownership during an SLO event
**Support/CS (Incident Driver for customer comms)**
- Owns the customer thread, updates, and expectation management.
- Owns the case record: timeline, impact, and customer confirmations.
- Initiates escalation and ensures correct teams are engaged.

**SRE On-Call (Technical Incident Lead)**
- Confirms whether the SLO is breached per the canonical evaluator.
- Drives mitigation and coordinates cross-functional response.
- Provides internal/external technical summary for comms.

**Infra/Fleet On-Call (Capacity / shape / provisioning owner)**
- Addresses capacity constraints, noisy-neighbor isolation issues, node failures, and shape-level performance anomalies.

**Observability (as-needed)**
- Validates metric correctness (e.g., streaming accounting anomalies), dashboard discrepancies, missing tags.

**Product Dedicated (as-needed)**
- Confirms entitlement interpretation (tier rules, burst policy, failover posture).
- Aligns on customer-facing messaging for edge cases.

**Finance/FP&A + Legal (as-needed for credits/comp)**
- Approves service credits above policy thresholds and any non-standard compensation.

## Severity levels (Support/CS view)
Use these severities to standardize response; SRE may map to their incident severities.

**SEV-1 (Critical, likely customer-impacting breach)**
- Premium tier customer reports sustained throughput below floor OR SRE burn-rate alert is firing.
- Clear production impact (e.g., queue buildup, timeouts, user-visible latency spikes).
- Immediate paging required.

**SEV-2 (High, likely degradation / at-risk of breach)**
- Throughput is below expected but not yet confirmed as an SLO breach.
- Customer impact is moderate OR limited to a subset of traffic/routes.
- Paging may be required depending on tier and duration.

**SEV-3 (Moderate, investigation / metrics discrepancy)**
- Conflicting metrics (customer vs Redwood dashboards), short-lived dips, or suspected measurement artifacts.
- No clear production impact.

## Paging and escalation triggers (when to page SRE / Infra)
### Page SRE On-Call immediately when any of the following is true
- Premium tier customer: reported throughput below committed floor for **>10 minutes**, AND they confirm they are not client-throttled.
- Any tier: a Dedicated Throughput SLO **burn-rate alert** is firing for the deployment.
- Any tier: repeated or escalating customer timeouts/5xx correlated with throughput drop.
- Any tier: throughput drop correlates with recent Redwood changes (rollout, routing, driver/runtime update).

### Page Infra/Fleet On-Call when any of the following is true (often in parallel)
- GPU/node health signals indicate failures, evictions, ECC errors, or repeated container restarts.
- Saturation: GPU utilization pinned while throughput drops (suspected contention, thermal throttling, network bottleneck).
- Capacity: the deployment appears mapped to the wrong pool/shape or lacks the expected headroom.
- Region-wide capacity constraint impacting Dedicated pools.

### Engage Observability (not a page; async unless SEV-1)
- Throughput dashboards disagree with billing/export numbers.
- Recent metric changes (e.g., streaming token accounting) suspected.
- Missing labels (deployment_id, model, region) prevents accurate attribution.

## Triage checklist (Support/CS)
Perform steps 1–8 before (or while) paging; do not block paging for SEV-1.

### 1) Confirm customer context (capture in ticket)
Collect:
- Deployment ID(s)
- Customer name + environment (prod/stage)
- Region
- Tier (Standard / Plus / Premium)
- Model + version (exact identifier)
- Shape (if known)
- Time window of impact (with timezone)
- Symptoms (queueing, latency, timeouts, functional failure)

### 2) Confirm the customer is driving enough load
Ask:
- Current request concurrency
- Any client-side rate limiting (explicit QPS cap, worker pool limit)
- Any recent application deploy or traffic change

If the customer is under-driving load, throughput may be low without being an SLO failure.

### 3) Validate “eligibility” red flags quickly
- Context lengths: are they within contracted max context / recommended ranges?
- Parameters: extreme max_tokens or non-streaming vs streaming changes?
- Model/version pin: did they change models recently?

### 4) Check the Dedicated Throughput dashboard for that deployment
Look for:
- Rolling-window throughput vs committed floor
- Error rate/timeout correlation
- GPU saturation, queue depth, admission control shedding
- Recent routing pool changes or failovers

### 5) Check if this is a metrics discrepancy
If customer numbers differ from Redwood:
- Ask what they are measuring (wall-clock end-to-end, tokens counted, streaming vs non-streaming).
- Request a single representative request ID and a 10–20 minute sample window.

### 6) Determine if the issue is isolated or systemic
- One deployment vs multiple
- One region vs multiple
- One model family vs multiple

### 7) Mitigation options you can offer immediately (tier-dependent)
These are customer-facing actions CS/Support may propose, with SRE confirmation where possible:
- Temporarily reduce max context / max output tokens (if within acceptable product behavior).
- Temporarily shift traffic to the contracted primary route only (avoid experimental routes).
- Enable/disable streaming based on their client constraints (note: streaming affects perceived latency but should not invalidate throughput).
- If the customer has multi-deployment setup: fail over to a secondary deployment (Premium: may be Redwood-assisted).

Do not promise failover across regions unless the tier/contract explicitly includes it.

### 8) Open an incident record (internal) if SEV-1/SEV-2
- Create/associate the internal incident ticket.
- Add the customer support ticket link.
- Add a running timeline (timestamps for all key events).

## Warm pool / startup considerations
Some tiers include warm pool guarantees (or faster recovery expectations). During triage, identify if the customer is within a warm-up window:
- Did the deployment recently scale from zero?
- Was there a recent model reload, config change, or node replacement?
- Did traffic resume after prolonged inactivity?

If warm-up is suspected, note it explicitly in the customer thread and ask SRE to confirm whether the event is excluded by policy.

## Customer communications guide
### Communication principles
- Use a single owner for customer updates (CS/Support), even if multiple engineering teams are engaged.
- Acknowledge impact quickly; avoid speculating on root cause.
- Provide time-bound next update (“next update in 30 minutes”).
- For Premium accounts, default to more frequent updates and written summaries.

### Target update cadence (recommended)
- SEV-1: initial response within 15 minutes; updates every 30 minutes until stable.
- SEV-2: initial response within 1 hour; updates every 2–4 hours until stable.
- SEV-3: initial response within 1 business day; updates daily or as material changes occur.

## Service credits / compensation (internal guidance)
**Important:** Do not commit to a credit amount in writing until approvals are complete.

### Which tiers are typically credit-eligible?
- **Premium:** written throughput commitment; credit review is standard for confirmed breaches.
- **Plus:** may be eligible depending on contract addenda; treat as “review required.”
- **Standard:** generally best-effort; credit requests require Product + Finance approval.

### When to initiate a credit review
Initiate when:
- SRE confirms an SLO breach per canonical evaluator, AND
- The breach is not excluded (e.g., customer throttling), AND
- The customer requests credits OR the account team recommends proactive review.

### Approval path (default)
1) CS/Support compiles incident details and SRE confirmation.
2) SRE provides breach window + summary.
3) Product Dedicated confirms entitlement interpretation (tier/exclusions).
4) Finance (Laura Bennett / delegate) validates policy compliance and amount.
5) Legal (Sofia Mendes / delegate) reviews if there is contract ambiguity or if a non-standard remedy is proposed.
6) CS communicates outcome to customer.

### What to include in the internal credit request
- Customer + contract/tier
- Deployment ID(s)
- Breach window (start/end in UTC)
- Evidence links: dashboard screenshots/links, evaluator output
- Customer impact summary (what broke, business impact if shared)
- Exclusion checks performed (throttling, workload constraints, warm-up)
- Proposed remedy type (service credit vs other)

## Evidence and logging requirements
For any SEV-1/SEV-2 and any credit review, capture:
- Customer ticket link
- Incident link (if created)
- Dashboard links (throughput vs floor, saturation, errors)
- Request IDs / traces used for debugging
- Summary of customer-side traffic confirmation
- Timeline of mitigations and when throughput stabilized

## Common failure modes and quick routing hints
- **Customer sees lower throughput than Redwood reports:** often a measurement mismatch (streaming token accounting, end-to-end latency vs throughput, counting input tokens). Engage Observability if persistent.
- **Throughput drops with GPU pegged:** suspect contention, node health, thermal throttling, network bottleneck; page Infra.
- **Throughput drops with low utilization:** suspect admission control limiting, customer throttling, or workload shift to longer contexts; page SRE; ask customer for concurrency/workload details.
- **Regional capacity constraint / failover behavior:** check tier failover posture; page Infra and SRE; do not promise cross-region without tier/contract confirmation.

---

## Post-incident expectations
For confirmed Premium-tier breaches (and significant Plus-tier events), request:
- SRE-written incident summary within 5 business days.
- Follow-up action items tagged to owning teams (SRE/Infra/Observability/Routing).
- If credits were issued: ensure the billing/metering record reflects committed vs burst usage correctly.

---

## Appendix: Minimal internal handoff note (for paging)
When paging SRE/Infra, include:
- “Dedicated Throughput SLO: suspected/confirmed”
- Deployment ID
- Tier + region
- Time window and whether ongoing
- Customer-confirmed load/concurrency (if available)
- Dashboard link
- Any correlated symptoms (timeouts, 5xx, queue depth)

This reduces back-and-forth and speeds mitigation.
