# Sup 28463 Double Charged For Retries

Source type: jira
Document ID: dsid_2f1f0358b85148f19d38f619891bfb15
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Acme AI reports being double-charged tokens on retry/fallback path (streaming) in Hosted API

Issue summary:
Acme AI reports that token usage on invoice preview is higher than their own aggregation of the Usage API for the same time window. They believe they are being billed twice when (a) their client retries after timeouts/5xx and/or (b) Redwood performs a server-side fallback retry during streaming.

Impact:
- Customer-reported overcharge risk and trust hit; potential credit request.
- Meter integrity risk: duplicate usage events on retry/fallback path may skew billing, pricing experiments, and gross margin model.

Customer evidence (from Support):
- Spike in billed tokens on 2026-02-06 around 19:00–21:00 UTC.
- Correlates to elevated 502/timeout errors and client retry behavior.
- Requests primarily via OpenAI-compatible /v1/chat/completions with stream=true.

Environment:
- Prod, region us-east
- Models: llama-3.1-70b primary, fallback variant enabled by routing policy

Suspected failure mode:
Usage event dedupe/idempotency for streaming retry/fallback path is insufficient. When an initial attempt emits partial tokens and then triggers an internal retry/fallback (or when the gateway retries before completion), both attempts may be marked billable and included in the billing aggregation, resulting in double-counting for a single logical client request.

2026-02-09 08:06 PT — Cole Summers (Billing Hooks) — Root cause found: on stream retry/fallback, we emit a usage event on the first attempt when the stream aborts, then emit another usage event on the retry attempt. Downstream billing aggregation does not enforce "single billable event per (account_id, idempotency_key, api_operation)" and treats both as billable. We should mark the first attempt as non-billable (or billable=false, attempt_status=aborted) and ensure the final successful attempt is the only billable event under idempotency.
Plan: ship fix under PR pr-27463 and validate with shadow billing compare (pr-27518) + perf-canary scenario update (pr-912).
