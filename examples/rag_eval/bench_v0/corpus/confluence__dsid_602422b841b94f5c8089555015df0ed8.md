Metering → Invoice Reconciliation Playbook (Hosted API + Dedicated)

## Purpose
This playbook defines how Redwood Inference reconciles customer invoices back to underlying metering data.

It is designed to:
- Detect billing-impacting metering issues early (daily/weekly) and before invoices are sent (month-end).
- Provide a repeatable, audit-ready trail from: **request logs → usage events → billing ledger → invoice lines → customer usage exports**.
- Standardize discrepancy classification and remediation (forward fixes, credits, backfills) with clear sign-offs.

**Applies to:**
- Hosted API (tokens/requests, embeddings, reranking)
- Dedicated (GPU-seconds)

**Not covered:** tax, FX, payment collection, revenue allocation. For month-end close controls, see *Month-End Close: Usage Reconciliation Checklist*.

---

## Definitions (canonical)
**Billing Ledger:** The internal finance-grade table used to generate invoice line items. This is the **source of truth for invoicing** once the billing freeze occurs.

**Usage Events:** Telemetry events emitted per request (or per logical request segment) that contain metered quantities (e.g., billed_tokens, requests, cache_hit attribution, gpu_seconds).

**Customer Usage Export / Usage API:** Customer-facing aggregation derived from usage events and/or ledger with documented lag and schema. Used for customer validation and dispute resolution.

**Reconciliation Grain:**
- Hosted API: account_id × usage_date (UTC) × region × model × product_route (chat/completions/embeddings/rerank) × price_plan
- Dedicated: account_id × usage_date (UTC) × pool_id × region × sku

**Freeze Window:** A time after which we stop changing ledger totals for the close period except via an approved adjustment process. (See Finance Slack thread and the month-end checklist.)

---

## Source-of-truth hierarchy
Use this hierarchy to decide which number “wins” in an investigation:
1) **Invoice PDF / issued invoice** (immutable customer artifact)
2) **Billing Ledger for the invoiced period** (invoice derivation) 
3) **Ledger reconciliation export** (Finance reconciler-friendly view of #2)
4) **Usage Events (raw + deduped)** (root cause and forward-fix validation)
5) **Customer Usage API/Exports** (customer observability; may lag)

If customer exports differ from ledger, treat it as either (a) a known lag/definition difference, or (b) a product bug requiring customer communication and a fix.

## Cadence
### Daily (operational)
Goal: catch pipeline regressions before they reach customer invoices.
- Run automated sanity checks (freshness, duplicates, null dimensions, drift).
- Triage any alert within 30 minutes during business hours.

### Weekly (control)
Goal: verify stability and trend anomalies.
- Reconcile previous 7 days ledger vs usage events.
- Sample 5–10 accounts across tiers and 2–3 regions.

### Month-end (close)
Goal: produce close-ready evidence package.
- Follow the Month-End Close checklist.
- Confirm freeze window and late-event policy.

## Thresholds and SLOs (default)
These are the default control thresholds. Adjustments require Finance + Telemetry Eng approval and must be documented.

### Data freshness
- Usage events availability lag: **p95 ≤ 15 min**, **p99 ≤ 60 min**
- Ledger export availability lag: **≤ 6 hours** (typical), **hard fail if > 12 hours**
- Customer usage export: **≤ 24 hours** (documented customer expectation)

### Correctness/drift
All drift thresholds are computed at the reconciliation grain.
- **Daily drift (Hosted API tokens):** absolute drift ≤ **0.25%** AND ≤ **$250** per account-day
- **Weekly drift:** absolute drift ≤ **0.10%** AND ≤ **$500** per account-week
- **Dedicated GPU-seconds:** absolute drift ≤ **0.20%** AND ≤ **$500** per account-day

### Dimension completeness
- Null rate for required dimensions (account_id, usage_date, product_route, region, model/sku): **< 0.05%**
- Duplicate usage_event_id rate post-dedupe: **< 0.02%**

## Step-by-step reconciliation procedure
### Step 0 — Start a reconciliation ticket and scope
Create a ticket (Finance Ops or shared tracker) containing:
- Period (UTC): start_date, end_date
- Product surface: Hosted API and/or Dedicated
- Data sources used (table/view names + snapshot timestamp)
- Whether any meter changes occurred in period (shadow billing, backfills, rollouts)

**Required attachment:** exported drift summary (CSV) from the approved reconciliation view.

---

### Step 1 — Verify freshness & completeness gates
Before comparing numbers, confirm we are not comparing incomplete data.

Checklist:
- Usage events ingestion is current (meets freshness SLO)
- Ledger export is current
- Any known incident windows are annotated (e.g., pipeline delays, regional outages)

If freshness is failing:
1) Pause reconciliation for the impacted window.
2) Page/notify the on-call owner per the meter sanity dashboard spec.
3) Re-run Step 1 after pipeline catch-up.

---

### Step 2 — Confirm dedupe/idempotency posture
Many “overbilling” reports are actually duplicate-event issues during retries/backfills.

Controls:
- Use **deduped usage events** view for aggregates (never raw events) unless investigating.
- Verify dedupe key includes request_id + attempt_id + idempotency_key (as applicable).

Common symptoms:
- Drift spikes aligned to deploys
- Drift concentrated in streaming endpoints
- Duplicate rate increases in one region/cluster

---

### Step 3 — Aggregate usage events to the reconciliation grain
Aggregate usage events to the same grain as the ledger/invoice lines.

**Hosted API:**
- Sum billed_prompt_tokens, billed_completion_tokens, billed_tokens_total
- Count billable_requests
- Track cache_hit tokens and non-cache tokens separately (for attribution/experiments)

**Dedicated:**
- Sum gpu_seconds_billable
- Separate reserved vs burst usage if applicable

Example (pseudo-SQL):
```
SELECT
  account_id,
  DATE_TRUNC('day', usage_ts_utc) AS usage_date,
  region,
  model,
  product_route,
  price_plan,
  SUM(billed_tokens_total) AS evt_billed_tokens,
  SUM(billable_requests) AS evt_requests,
  SUM(cache_hit_tokens) AS evt_cache_hit_tokens,
  COUNT(*) AS evt_rows
FROM usage_events_deduped
WHERE usage_ts_utc >= :start_ts AND usage_ts_utc < :end_ts
GROUP BY 1,2,3,4,5,6;
```

**Important:** ensure timezone alignment. All reconciliation is done in **UTC** unless explicitly performing a customer-specific export in their billing timezone.

---

### Step 4 — Pull ledger totals for the same grain
Use the approved ledger reconciliation export (preferred) or ledger table (restricted).

Confirm:
- Ledger includes the same dimension keys (or a documented mapping)
- Price plan changes mid-period are represented consistently

Example (pseudo-SQL):
```
SELECT
  account_id,
  usage_date,
  region,
  model,
  product_route,
  price_plan,
  SUM(billed_tokens_total) AS led_billed_tokens,
  SUM(billable_requests) AS led_requests,
  SUM(amount_usd) AS led_amount_usd
FROM billing_ledger_recon_export
WHERE usage_date BETWEEN :start_date AND :end_date
GROUP BY 1,2,3,4,5,6;
```

---

### Step 5 — Compute drift and classify discrepancies
Join usage-event aggregates to ledger aggregates and compute:
- Token drift: (evt_billed_tokens - led_billed_tokens)
- Request drift
- Amount drift (if ledger contains pricing)

Classification (pick one primary label):
1) **Late-arriving usage events** (events present after ledger cut; freshness breach)
2) **Duplicate events / idempotency regression** (evt > ledger, duplicates elevated)
3) **Retry/fallback double-count** (streaming + server retry; known failure mode)
4) **Missing dimensions** (region/model/route nulls causing mis-bucketing)
5) **Token definition mismatch** (raw vs billed tokens; tool calls/structured output)
6) **Embeddings batch accounting bug** (per-input vs per-request counting)
7) **Timezone boundary mismatch** (customer export vs ledger day boundaries)
8) **Expected business rule difference** (e.g., free tier, minimums, rounding rules)

Decision rule:
- If drift is **within thresholds** and no systematic bias is observed: record as pass.
- If drift is **outside thresholds**: proceed to Step 6.

---

### Step 6 — Cross-check against customer usage API/export
For customer disputes and customer-visible consistency, compare ledger totals to the customer usage export/API.

Procedure:
1) Pull usage export for same account and UTC period.
2) Confirm export schema version and whether it shows **billed_tokens** vs **raw_tokens**.
3) Reconcile at daily grain, then drill down by model/region/route.

If customer usage export differs from ledger:
- Determine whether it is due to **lag** (export behind ledger) or **definition mismatch**.
- If definition mismatch: document which field is correct for customer messaging and ensure docs/SDK are aligned.

---

### Step 7 — Drill-down playbook by failure mode
Use this section when Step 5 flags drift outside thresholds.

#### A) Late-arriving events
Signals:
- Freshness SLO breach
- Drift concentrated in the most recent 24–48 hours

Actions:
- Re-run reconciliation after pipeline catch-up.
- If ledger has already frozen for the period: do **not** change ledger without CFO/RevRec approval.
- Add incident annotation and attach evidence (lag chart + impacted partitions).

#### B) Duplicate events / idempotency regression
Signals:
- Duplicate rate increases
- Drift correlates with partial failures/backfills

Actions:
- Identify duplicate key collisions and affected endpoints.
- Engineering to confirm dedupe window and idempotency key behavior.
- Prefer forward fix + backfill usage events view; do not retroactively re-issue invoices without approval.

#### C) Retry/fallback double-count (Hosted API)
Signals:
- Drift spikes on streaming/chat
- Higher variance in routes using fallback routing

Actions:
- Validate whether both attempts emitted billable events.
- Confirm request_id/attempt_id relationship in logs.
- If confirmed: implement fix + run shadow billing compare before enabling.

#### D) Missing/incorrect dimensions (region/model/route)
Signals:
- Null dimension rate > threshold
- Drift appears as “missing” in one bucket and “extra” in another

Actions:
- Determine whether ledger applies defaults/mappings that usage events do not.
- Backfill dimension fields if needed (with Finance approval if it changes customer exports).

#### E) Token definition mismatch (raw vs billed)
Signals:
- Customer compares invoice to raw token counts from SDK logs
- Tool calls / structured output present

Actions:
- Confirm invoice uses **billed_tokens** per meter spec.
- Ensure customer usage export includes billed_tokens and is documented.
- If mismatch is due to OpenAI-compatible endpoint semantics: coordinate with Product + Docs.

#### F) Embeddings batch accounting
Signals:
- Drift only on embeddings route
- Customers using batched input arrays

Actions:
- Validate whether per-input tokenization is applied consistently.
- Confirm “billable unit” definition in meter spec.
- Run unit tests / regression scenario where batch sizes vary.

#### G) Timezone boundary mismatch (exports)
Signals:
- Customer sees day-over-day spikes at midnight local time

Actions:
- Confirm invoice period is UTC.
- If customer requires a local-time export, provide it as an informational report but do not change invoice basis unless contract specifies.

---

### Step 8 — Decide remediation path (forward fix vs adjustment)
Use this decision tree once root cause is identified.

**Default policy:** prefer **forward fixes** with clear documentation and audit trail. Avoid retroactive invoice changes.

Remediation options:
1) **No action** (variance within threshold; document)
2) **Forward fix only** (metering bug fixed; applies to future usage)
3) **Customer credit** (invoice remains; credit memo or usage credit issued)
4) **Invoice adjustment / re-issue** (rare; requires CFO + Legal)
5) **Backfill** (usage events/customer export corrected; ledger may or may not change depending on freeze)

Minimum documentation for any remediation:
- Root cause summary
- Impacted period + accounts
- Quantified $ impact (best estimate + bounds)
- Customer comms plan (Support/Sales owner)
- Approvals captured (names + date)

---

### Step 9 — Produce an audit-ready evidence package
Attach to the reconciliation ticket:
- Drift report (by account/day/model/region/route)
- Freshness and completeness charts for the period
- Any incident annotations
- Query fingerprints (table/view names + snapshot timestamp)
- Final sign-off comment (RevRec + Eng)

Retention:
- Store evidence per retention policy; do not export raw request logs.

## Appendix A — Minimal reconciliation checklist (quick pass)
Use for weekly spot-checks.
1) Confirm freshness SLO pass
2) Run ledger vs usage events drift (last 7 days)
3) Sample 10 accounts (2 enterprise, 5 mid-market, 3 self-serve)
4) Validate top 5 models per account
5) Confirm null rate and duplicate rate are under thresholds
6) Record pass/fail and open investigations

---

## Appendix B — Customer dispute workflow integration
When Support/Sales opens a billing dispute:
1) Require: invoice id, account id, disputed period, customer’s comparison method (SDK logs vs export)
2) Run Step 6 first (customer usage export alignment)
3) If discrepancy persists, proceed Steps 3–5 at a narrower scope
4) Provide customer-facing explanation using approved language (Legal playbook)
5) If credit is recommended, route for approval per sign-off rules

---

## Appendix C — Known recurring failure modes (historical)
- Streaming retry/fallback double-counting when both attempts emitted billable events
- Missing region dimension leading to mis-priced regional rates and margin attribution errors
- Embeddings batched requests counted per-request instead of per-input tokenization
- Customer export timezone misunderstanding (UTC invoice vs local-time dashboards)

---

## Change control
Any change to metering definitions or ledger derivation logic must follow:
- Shadow billing compare for at least one full business cycle (typically 7 days)
- Documented before/after diffs and expected variance
- Finance/RevRec sign-off prior to enabling invoice-impacting behavior

(See the metering change control process ticket and the meter sanity dashboard spec for alerting requirements.)
