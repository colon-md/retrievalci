# Sup 28451 Invoice Tokens Do Not Match Usage Api

Source type: jira
Document ID: dsid_d04dea2d736445c3af6934b900a12ef9
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Invoice token counts do not match Usage API export (Acme AI) — suspected duplicate metering on streaming retry/fallback

Customer: Acme AI

Issue summary
- Acme AI reports February invoice tokens are higher than their Usage API export totals for the same billing period.
- They are summing Usage API daily exports by model and region and comparing to the invoice line items.

Impact
- Potential overcharge claim / invoice dispute.
- Risk: RevRec confidence + customer trust.

Billing period / artifacts
- Invoice: INV-2026-02-ACME (Feb 2026 usage)
- Customer comparison window: 2026-01-01 00:00:00 UTC through 2026-01-31 23:59:59 UTC (per invoice)
- Customer export method: /v1/usage?start_date=2026-01-01&end_date=2026-01-31 (aggregated by day)

Observed discrepancy (initial)
- Invoice total billed tokens: 9,842,110
- Usage API export total tokens: 9,521,774
- Delta: +320,336 tokens on invoice (+3.36%)
- Delta concentrated in streaming chat completions on rw-llama-70b-instruct (us-east) between 2026-01-18 and 2026-01-24.

Hypotheses
1) Duplicate billing ledger writes on server-side retry/fallback for streaming requests (same customer request counted twice).
2) Usage API export is de-duping by request_id while billing ledger is not (or uses different idempotency key).
3) Late-arriving events / backfills included in invoice but missing from Usage API due to export freshness window.
4) Token counting mismatch between OpenAI-compatible endpoints vs native endpoints (less likely for this specific window).

Investigation notes (what we checked)
- Compared billing_ledger_v1 aggregates vs usage_events_v1 aggregates for the same account_id, model, region, and day.
- Sampled 25 request_ids from the highest-delta day (2026-01-21) and traced request lifecycle in request logs.
- Looked for patterns: streaming disconnects, 5xx on first attempt, fallback routing to a compatible variant, and subsequent successful retry.

Findings
- Confirmed duplicate ledger rows for a subset of streaming requests that experienced a server-side retry/fallback.
- Pattern:
  - First attempt returns partial stream then terminates (upstream timeout / capacity blip).
  - Gateway triggers internal retry to a compatible backend.
  - Billing hook emitted two chargeable usage records (same customer idempotency key missing/empty on first attempt) and both were ingested into billing_ledger_v1.
  - Usage API export (customer-facing) is sourced from usage_events_v1 where we dedupe on (account_id, request_id, event_sequence) and therefore only shows one.
- This explains invoice > usage export.

Evidence (redacted)
- Example request_id: req_01HNX9V8T1Q2A3B4C5D6E7F8
  - usage_events_v1: 1 record, total_tokens=3,412
  - billing_ledger_v1: 2 records, total_tokens=3,412 and total_tokens=3,412 (duplicate)
  - gateway log: "stream_retry=true" and "fallback_route=rw-llama-70b-instruct@v2026-01-15 -> rw-llama-70b-instruct@v2026-01-15" (same model, different backend instance)

Customer communication needs
- Provide a clear explanation: discrepancy is a metering bug on retry/fallback for streaming, not a difference in tokenization rules.
- Provide corrected totals for the disputed period and proposed remediation (credit vs re-issue) per Finance/Legal policy.

Proposed remediation
- Engineering: ship fix to ensure billing hook idempotency for streaming retry/fallback (prevent duplicate ledger entries) + add regression tests.
- Finance: issue credit for confirmed overbilled tokens for Jan usage (applied on next invoice), avoid retroactive re-issue unless required.
