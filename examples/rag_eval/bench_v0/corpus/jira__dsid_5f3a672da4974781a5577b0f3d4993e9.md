# Sup 864999 Sliced Ingest Priority Class Throttle Config

Source type: jira

Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Sustained 429s during sliced ingest with priority class mismatch after dedicated pool migration

Issue summary:
Customer reports sustained 429 responses across a set of sliced ingest workers after migrating ingestion to a Dedicated pool. The traffic mix is: low-latency chat calls (priority=high) and large multi-part embedding uploads (priority=background). After the migration, background uploads are intermittently exhausting burst credits at the per-route layer and causing opaque 429s for high-priority short calls during peak windows.

Impact:
- ~30% of short chat requests returned 429 between 09:00-11:30 PDT on 2026-03-10 for Streamly AI.
- Customer reports user-facing errors and retry amplification in their orchestrator.
- Business impact: partial degradation of their real-time features during peak ingest windows.

Environment:
- Dedicated pool: dp-132-usw
- Region: us-west
- Ingest coordinator version: 2026.02.3
- Rate-limiter service version: 1.14.0-canary

Observed behaviour:
- Persistent 429s reported by customer for short-lived chat requests coincident with high concurrency uploads.
- Throttle headers (when present) show per-route-window exhaustion; some requests show no X-RateLimit headers (opaque 429).

Expected behaviour:
- High-priority interactive calls should be isolated into a small burst pool and not experience 429s while background uploads borrow burst credit from a separate pool (or be rate-limited gracefully).

Customer asks:
- Why are small chat calls getting 429s when their load is relatively small?
- How to configure priority classes and burst pools for dedicated pools to prevent this?
1) Use the customer's traffic shape: 100 concurrent short chat requests (avg 50ms) + 20 parallel multipart embedding uploads (avg 45s each).
2) Route both traffic shapes through the dedicated pool dp-132-usw with current priority-class config (default: interactive=high, background=low).
3) Start background uploads in a tight window to simulate a scheduled flush (cron at 09:00).
4) Observe 429s on interactive calls between 09:00-09:30 when background uploads ramp up.

2) Rate-limiter metric rlim.route.tokens.available shows transient negative deltas when background workers register multiple concurrent signed-url callbacks (double-counting observed in edge proxy earlier; needs validation).
3) Canary release 1.14.0 has a code path that recalculates burst-pool weights when a route attaches to a new pool; race condition can allow an attach to read stale bucket state and permit borrow that appears to zero out another pool's credits.
4) Some 429s are opaque because edge proxy returns before rate-limiter headers are populated when under heavy callback fanout. Need to instrument edge path to surface X-RateLimit headers even under backpressure.
5) No evidence of dedicated pool GPU exhaustion; autoscaler metrics are stable.
1) Short-term (apply during call): adjust per-route burst allocation to reserve 20% of interactive burst credits exclusively for priority=high routes on dp-132-usw.
2) Short-term: recommend customer stagger flush windows and add randomized jitter to callback fanout.
3) Medium-term: devs to backport fix to rate-limiter that prevents borrow from reading stale burst-pool state; target patch 1.14.1.
4) Medium-term: instrument edge proxy to always emit X-RateLimit headers even when returning 429 due to callback fanout backpressure.
