# Pr 489 Add Restore Apply Workflow And Validation

Source type: github
Document ID: dsid_b8691ba2e29542838afae93928ad816a
Release: v1.0.0
Benchmark: EnterpriseRAG-Bench synthetic upstream data
Add `restore apply` workflow with preflight checks and post-restore validation report

Implements the end-to-end restore path for Redwood Private in the installer CLI. This adds a new `restore apply` command that: (1) performs strict preflight checks (cluster connectivity, namespace/CRD readiness, version compatibility, storage backend access, and integrity checks on the backup artifact), (2) restores control-plane state in a deterministic order (cluster prerequisites, control-plane chart values/config, DB restore, then service rollout), and (3) runs post-restore health checks and emits a validation report that Support/SRE can use for escalation. The goal is to make restores repeatable, safer to run under time pressure, and easier to debug when something goes wrong. Restore ordering is intentionally conservative to avoid the on-prem issues we saw in SUP-18477 (missing CRDs/incorrect bootstrap order). Security note: we do not restore secret material from backups; we restore secret references and validate that required KMS/external secret providers are reachable. The report includes sanitized environment details (versions, timestamps, artifact IDs), step-by-step results, and pointers to relevant logs. Usage examples: `private-installer restore apply --artifact s3://<bucket>/backups/<id> --namespace redwood-private --report /tmp/restore-report.json` and `private-installer restore apply --artifact ./backup.tgz.enc --offline --validate-only`.
- Private installer: introduce `restore apply` with deterministic ordering, guardrails, and a support-ready validation report. - Adds optional `--validate-only` mode and clearer failure categorization (preflight/artifact/restore/validation).
