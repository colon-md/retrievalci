chat v1: parameter normalizer, role aliasing, stop dedupe and streaming backpressure improvements for /v1/chat.completions parity

Motivation: Close a set of subtle incompatibilities between our /v1/chat.completions adapter and OpenAI's behavior that have been causing developer friction in streaming function-call flows and stop/logprobs handling. This PR implements a compact 'parameter normalizer' layer and a streaming framing/backpressure handshake to make the adapter tolerant of real-world clients and OpenAI-style payloads. Summary of changes: - Parameter normalizer: unify and sanitize incoming params (max_tokens, temperature, top_p, n, presence_penalty) and map them to our internal API consistently; accept both string and numeric encodings, preserve explicit nulls, and prefer client-provided values for edge cases. - Role aliasing: accept and normalize role names coming from clients ("assistant", "system", "user", "tool", and legacy aliases like "ai" or "bot"); ensure role is preserved in both final and streaming envelopes. - Stop sequence deduplication: support stop as string or array; expand single-string stops into an array; dedupe overlapping sequences (e.g. ["\n\n", "\n\n\n"]) and maintain order stability to match OpenAI semantics. - Sparse logprobs: add an opt-in sparse_logprobs mode that mirrors OpenAI's sparse token-level logprob structure when requested; keep heavy-weight computation behind an explicit flag to avoid perf regressions. - Streaming framing & backpressure: introduce a light handshake in the stream start (sent as first SSE/delta) that communicates our token framing and heartbeat expectations; implement backpressure signaling for HTTP/2 clients to avoid partial JSON objects when a client pauses mid-stream. - Function-call streaming support: improve chunking for function_call.name/arguments so that streaming deltas assemble deterministically into valid JSON payloads even across reconnects; final chunk action inserts a stable end-of-function marker to make reconstruction robust. - Response envelope parity: normalize streaming deltas to include a minimal structured 'choice' envelope (id, index, finish_reason) so downstream OpenAI-compat consumers don't need ad-hoc heuristics. - Tests & docs: add unit and integration tests for the normalizer, streaming handshake scenarios, and a new example in the docs showing a robust client reconnection strategy. Checklist: - [x] Parameter mapping implemented and unit-tested - [x] Streaming handshake and backpressure implemented with integration tests - [x] Role aliasing covered by parser tests - [x] Stop dedupe logic covered and fuzz-tested - [x] Sparse logprobs gating and tests - [x] Docs updated (compatibility section + example client) - [x] Runtime perf smoke tests run on canary cluster Notable design decisions: - We intentionally do not try to replicate obscure historical OpenAI quirks; instead the normalizer applies conservative, well-documented rules and exposes a strict compatibility flag for customers who really need historical behavior. - The streaming handshake is opt-out via a client header (X-Redwood-Compat: openai-v1) to avoid breaking existing non-OpenAI clients. - Sparse logprobs are computed on-demand and annotated in the response metadata to allow downstream billing/observability.
